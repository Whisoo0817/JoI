## Abstract (flow)
• [PROBLEM] end-user reactive automation이 자연어로 이동 중이고, on-device sLLM(≤9B)이 privacy/latency상 유일하게 viable한 generator — 하지만 생성된 rule-code를 *배포 전에 사용자 의도 대비 검증할 수 없다*.
• [기존 검증이 안 옮겨지는 이유] NL→code 검증은 *oracle*이 있을 때 작동함: 실행 비교용 gold 결과나 test/example suite, 또는 정답을 정의하는 human-authored property/monitor. reactive automation은 배포 시점에 이 중 무엇도 없음: output이 *이벤트열에 걸친 미래 device side-effect*라 on-demand 비교 불가; idiomatic rule 코드가 *temporally opaque*하고 신뢰성 있게 inspect 불가(edge-vs-level·재무장이 표면에 없음; LLM으로 NL 되읽기는 충실성 보장 없고 executable oracle도 안 나옴); 그리고 *비개발자 end user는 temporal 형식 spec을 authoring할 수 없음*.
• [아이디어 — hero contribution: IR-as-spec] reactive-temporal automation의 scoped fragment를 위한 *normalized behavioral representation* **Timeline IR**을 도입: 8개 typed step operator(+ field-level 확장)로 그 fragment의 reactive 구성요소(P1–P14)를 target-code idiom이 아닌 typed field로 표현 → 여러 valid JoI 인코딩에 걸쳐 confirm된 behavior가 안정적. IR은 NL로 deterministic 렌더돼 **사용자가 승인(confirm-by-approval — 형식지식 불필요, 비개발자 smart-home user 눈높이)**; confirm 이후 **IR이 곧 spec — 도메인에 없던 oracle을 instantiate**. 거기서 **reference trace(IR-SIM)**와 **construct-derived boundary obligation(IR-FSM)**을 derive해, **LLM-free verifier가 후보 JoI를 confirm된 IR 대비** *bounded trace conformance*(Rung 1)로 검증 — trace 비교를 ad hoc diff가 아닌 구조적·진단가능한 검사로 만듦.
• System: ≤9B on-device 모델 위의 **decomposed NL→IR→JoI 파이프라인** — intent analysis, service & device mapping, IR assembly, lowering, **매 join에 deterministic glue(autonomous agent 아님; LLM-free verdict)** — + two-projection verifier (IR-SIM trace oracle, IR-FSM obligation oracle)가 **명시된 observation model 하의 bounded behavioral conformance checking**을 수행 → deterministic accept/reject + retry 구동. (codex: positive contract framing, "never sound acceptance" 아님.)
• Results headline [DATA]: E2E accuracy vs (a) cloud-direct, (b) cloud self-repair, (c) **fine-tuned-SLM baseline**; verifier rejection power (mutation-adequacy [DATA]%), construct-boundary coverage [DATA]%; real-hub deployment [DATA]; user study — 사용자가 raw code 대비 IR에서 intent error를 [DATA]% 더 잡음.
• Contribution framing: LLM이 생성한 reactive automation을 *user-confirmed, 비개발자가 승인 가능한 spec 대비 behaviorally verifiable*하게 만든 최초 사례 (on the edge) — 사람이 형식 spec을 authoring하지 않고도.

*(ARC — PROBLEM-FIRST 척추, 2026-05-26 재구조화. 순서: trust hook → reactive-temporal beyond TAP → 코드는 non-canonical → correctness는 behavioral(trace)이어야 → behavior엔 oracle 필요한데 reactive엔 없음(missing-oracle) → 기존 work는 우회 → raw trace는 confirm artifact로 부적합 → user-confirmed IR에서 oracle derive(confirm과 verify 분리). • **Beat 1 — trust hook (locked opening sentence) + edge 맥락**: *"A home hub can now turn a sentence like 'whenever the door opens after 10pm, turn on the hallway light for two minutes' into automation code locally — but neither the user nor the hub can tell whether the resulting reactive rule actually matches that intent."* (NL automation + edge/local + reactive-temporal + trust를 한 줄에.) 왜 local인가: privacy + latency + offline ⇒ generator는 on-device 소형 모델(≤9B)에서 동작해야 함 [CITE GPIoT, LLMind2.0]; 그 scale에서 intent+timing+selector+idiom을 동시 합성하는 single-pass NL→rule-code는 자주 실패 [DATA 9B-direct] → 배포 전 correctness check가 선택이 아님.
• **Beat 2 — reactive-temporal, beyond if-trigger-then-action**: [CLAIM] 현대 end-user automation은 one-shot trigger→action을 넘어 *reactive-temporal* behavior로 이동 — duration("5분 동안 열림"), delay, periodic/cron 스케줄, 반복 실행, cancellation/reset, persistent state, all/any quantification. intent는 단일 함수 출력이 아니라 *시간·이벤트에 걸친 behavior*.
• **Beat 3 — 코드는 canonical artifact가 아니다 (THE BUILDUP — 사용자 우선순위)**: [CLAIM] 이 reactive-temporal behavior는 플랫폼/DSL마다 *다르게* 인코딩됨 — 어떤 곳은 sustained 조건을 duration trigger(`for:`)로, 어떤 곳은 timer로, polling loop·state variable·delayed action·FSM transition으로; 각 플랫폼은 이 primitive의 *서로 다른 부분집합*만 first-class로 노출하고 나머지는 hand-written idiom으로 강제. 그래서 하나의 intent가 *여러 valid lowering*을 허용 [DATA: 하나의 intent → ≥3개 valid JoI encoding; N개 모델 family의 idiom 선택 분산 = X%], 표면상 비슷한 프로그램도 temporal edge case에서 갈림(3분 열렸다 닫히면 5분 timer가 reset되나? 1회 발동인가 반복인가?). **표현력 있는 reactive automation에서 코드는 따라서 canonical artifact가 아니다.** (Framing 가드: "DSL이 primitive 없음"도 "기존 dataset이 쉬움"도 아님 — 방어 가능한 주장은 *조사한 end-user/consumer RA-DSL 중 어느 것도 우리 P1–P14 fragment를 first-class-complete로 덮지 못함*(P6 hysteresis / P9 alternation은 어디서도 first-class 아님; industrial/synchronous 언어는 §10.7로 유보) → 생성 코드 non-canonical; §2.3에서 측정.)
• **Beat 4 — correctness는 behavioral이어야**: [CLAIM] 코드가 non-canonical이므로 명령당 단일 golden program은 무의미; correctness는 *behaviorally*, 즉 event/action trace 동치로 정의해야 함. **빠진 artifact는 golden *program*이 아니라 golden *behavior*.** (text-to-SQL/code-gen이 이미 한 이동 — string 동치 대신 결과 동치 / unit test — 을 reactive-temporal 출력에 적용.)
• **Beat 4.5 — concrete failure (코드 레벨에서 안 보이는 behavioral 버그)**: 그럴듯하지만 틀린 rule 하나. "whenever the door opens, turn on the light"를 *rising edge*(열릴 때마다 1회) 대신 *level* 체크(열려있는 동안 계속 발동)로, 또는 re-arm 안 되는 `triggered` latch(1회만 발동). 읽으면 멀쩡하고 실행도 됨 — 그러나 조용히 의도 위반. 기존 도구는 못 잡음: consumer DSL은 syntax만 검사, runtime trace는 버그가 이미 실제 device에 작동한 *후*에야 기록. 버그가 behavioral이라 behavioral check만 잡을 수 있음.
• **Beat 5 — behavior엔 oracle이 필요한데 reactive automation엔 없다 (missing-oracle problem)**: [CLAIM] trace-equivalence엔 비교할 *expected* trace가 필요. 후보를 실행하면 *actual* trace만 나옴 — 이 프로그램이 무엇을 하는지일 뿐, 사용자가 의도한 게 무엇인지가 아님 (**execution ≠ verification**). 다른 NL→code 도메인은 이 oracle을 제공: text-to-SQL은 gold 결과 대비, code-gen은 unit test — 둘 다 (i) 독립적 실행 reference와 (ii) 사용자가 확인 가능한 inspectable output을 가짐. reactive automation은 배포 시점에 둘 다 없음: output은 *미래 이벤트-트리거 device side-effect*("밤 10시 문 열릴 때"), *독립적 reference behavior 없음*, 일반 사용자가 exhaustive event/side-effect trace를 넘겨줄 거라 기대 못 함. 이것이 **missing-oracle problem**: free-form NL 입력, non-canonical reactive-code 출력, gold trace 없음. (더 깊게는, reference가 발화에 *잠재해 있지조차 않을* 때도 있음 — *"if the door opens"*는 one-shot vs recurring을 underspecify; confirmation만이 해소 [model 약점 아니라 원리적: §3.3b].) [§1에선 문제 + 한 가지 대조만; 전체 prerequisite 표와 execution≠verification 형식화는 §3.3.]
• **Beat 6 — 기존 work는 우회한다 (positioning, 공격 금지)**: [CLAIM] 기존 work는 missing oracle을 세 가정 중 하나로 우회하며, 그 어느 것도 NL-generated reactive automation엔 성립 안 함: (i) automation을 simple TAP 형태로 제한; (ii) *user-/expert-authored* formal property나 test oracle을 가정(safety/LTL — negative guardrail); (iii) NL-intent conformance가 아니라 *executability/structure/post-hoc behavior*를 검증. 특히 trace 기반 시스템은 trace를 **what happened**를 설명·수리하는 데 사용(post-hoc 디버깅 / anomaly / rule synthesis); **우리는 trace로 *what should happen*을, 배포 전에 검증.** (한 줄: *기존 work는 일어난 일을 설명·수리하려고 trace를 쓰고, 우리는 일어나야 할 일을 검증하려고 trace를 쓴다.*) [→ T1–T7 표 §2.5/§10]

## §3 The Spec Gap (problem formalization) (flow)
• §3.1 세 audience가 같은 질문 "맞나?"(user / system / auditor) — 누구도 oracle 없음.
• §3.2 NL이 spec일 수 없는 이유(모호성); 생성 코드가 spec일 수 없는 이유(idiom non-canonicality — 어떤 RA-DSL도 first-class-complete 아니라 idiom 항상 잔존; 그리고 비교할 *독립적* reference behavior 부재).
• §3.3 **[CLAIM, 기억에 남는 framing] 왜 reactive automation은 text-to-SQL의 검증가능성을 빌릴 수 없나.** NL→code 검증은 도메인이 oracle을 제공하는 곳에서 작동; reactive automation은 전제조건이 하나도 없음:
  | 자동검증 전제조건 | text-to-SQL / code-gen | reactive automation |
  |---|---|---|
  | intended behavior의 독립적 실행 *reference* | ✅ gold query 결과 / unit test | ❌ rule은 hub에서 돌지만 독립 reference behavior 없음 |
  | 사용자가 확인하는 inspectable output | ✅ 결과 행을 on-demand로 봄 | ❌ output = *미래 이벤트-트리거* device side-effect; on-demand 아님 |
  | 사용자가 줄 수 있는 I/O 예시 | ✅ PBE / test | ❌ behavior가 이벤트열에 걸친 temporal; exhaustive trace 기대 못 함 |
  | deploy-time reference | ⚠️ gold 필요(offline) | ❌ novel 명령, gold 없음 |
  • **Execution ≠ verification**: emulator는 후보를 실행(무엇을 하는지)하지만 oracle 아님(의도와 맞는지). 빠진 hard piece = *reference behavior* + *temporal input*.
• §3.3b **[CLAIM, oracle이 없는 가장 깊은 이유] reference는 NL만으로 추론 불가 — 빠진 정보는 model이 아니라 input channel에 있음.** (1) reactive-temporal 명령에서 intended behavior의 일부는 *발화에 entail되지 않음*: "if the door opens, turn on the light"는 one-shot vs recurring을 underspecify하고, "조금 있다 꺼"는 delay를 underspecify함. (2) generator가 적용하는 lexical convention("if"→one-shot, "whenever/every time"→recurring)은 **prior이지 truth가 아님** — 합리적 default reading일 뿐, *이* 사용자의 의도에 대한 보장이 결코 아님. (3) **더 크거나 좋은 model로도 안 닫힘**: scale은 *likely reading의 prior*를 sharpen할 뿐, 사용자가 발화에 넣지 않은 temporal 선택을 *관측*할 수는 없음 → 이는 capability gap이 아니라 input의 원리적 한계 ["그냥 더 큰 LLM 쓰면 되잖아" 선제 차단]. (4) 따라서 confirmation은 usability 편의가 아니라 **oracle을 instantiate하는 연산**: 하나의 underspecified 명령을 단일 executable reference spec으로 만들고, 그것이 바로 위 표가 결여한 reference임. (5) confirmation *이후에야* IR-SIM(reference trace) + scenario synthesis(temporal input)가 trace-equivalence를 decidable하게 만듦. [confirmation 지점은 생성 코드가 아니라 반드시 IR이어야 함 — §4, §6]
• §3.4 **왜 기존 검증으로 안 닫히나 (정밀 — "reactive automation은 검증 불가"가 아님; *테스트는* 됨)**: 활발한 IoT-검증 문헌(TAPInspector'22, TAPFixer'24, HAWatcher'21, Soteria/IoTSan)은 *human-authored* rule의 **safety / rule-interaction / anomaly** 속성을 검증; 구조 diagnostics(AWS IoT Events "Run analysis", HA schema, TaskSense DAG)는 well-formedness만. **trace 기반 시스템은 trace를 *반대* 방향으로 사용**: Trace2TAP / "Helping Users Debug TAP" / HAWatcher / HA-Trace는 *actual* 과거 trace를 소비해 **what happened**를 디버그·수리·마이닝·설명 — post-hoc, 이미 배포된 rule 위에서. **우리는 confirm된 spec에서 *expected* trace를 derive해 생성 코드를 그 대비 검사 — *what should happen*, 배포 전.** (한 줄: *기존 work는 일어난 일을 설명·수리하려고 trace를 쓰고, 우리는 일어나야 할 일을 검증하려고 trace를 쓴다.*) **Supporting precedent(위협 아님)**: expert test harness는 시나리오+assertion을 누군가 제공하면 reactive automation을 *테스트할 수 있음*을 증명 — 가장 강한 openHAB JRuby RSpec + `timecop`는 `for:`-sustain timing까지 offline assert — **단 assertion이 expert-authored; NL-generated automation은 배포 시점에 그게 없음.** **우리가 아는 한 누구도 T6 결합을 제공 안 함**: NL 명령 → *user-confirmed executable behavioral spec* → *auto-synthesized temporal scenario* → *LLM-generated idiomatic code*의 expected trace 대비 *trace-equivalence*. 다른 명백한 해법도 실패: code→NL round-trip(코드가 *temporally opaque* — edge-vs-level·재무장이 표면에 없음; LLM 되읽기는 부분적이고 *충실성 보장 없으며*(또 다른 ungrounded LLM → confirm 대상이 confirm 필요), trace-equivalence용 *executable oracle도 안 나옴*, NL 문자열만); model checking(φ·formal target semantics 없음, state explosion); monitor synthesis(hand-authored LTL). [전체 §10]
• §3.4b **[CLAIM] 왜 confirm 대상이 raw trace가 아니라 spec인가** (IR을 confirm하는 근거, §4/§5): oracle gap은 사용자에게 trace를 보여 승인받는 것으로 *닫히지 않음*. raw trace는 길고, 시간-인덱스, low-level이며, *하나*의 scenario만 보여주고 coverage 보장 없음 — 시각화(HA-Trace, TAP-debug history view)는 *what happened* 이해를 돕지만 비전문가가 edge case(sustain 경계·reset·one-shot-vs-repeat·all/any) 전반의 *full temporal behavior*를 certify하게 해주진 못함. 그래서 두 역할을 분리: 사용자는 *compact behavioral spec*(작고·읽을 수 있고·scoped IR fragment에 대해 complete)을 confirm하고, 검증에 필요한 *다수*의 trace는 — 사용자가 아니라 — 시스템이 그 confirm된 spec에서 derive해 처리.
• §3.5 [CLAIM] 빠진 조각 = NL-confirmable AND machine-checkable 중간 artifact — 거기서 도메인에 없던 *deploy-time reference oracle을 derive*하고, 거기서 시스템이 사용자가 읽을 필요 없는 검증 trace들을 합성. 나머지가 이를 구축.

## §4 Requirements for a Confirmable Behavioral Spec (flow)
• [CLAIM] gap을 닫는 artifact는 세 속성을 가져야: **Confirmable / Executable / Obligation-derivable**.
• [TAB] property → meaning → forced-by-failure → realized-as (C1 NL-projectable, C4 simulator-interpretable, C3 FSM-derivable).
• Idiom-invariance (C2) = 셋 모두를 여는 cross-cutting *principle*; sLLM-lowerability (C5) = deployment 제약이지 spec 속성 아님.
• [CLAIM] **Confirmable은 usability가 아니라 oracle formation을 위해 필요함**(§3.3b): 일부 temporal 선택(one-shot vs recurring, 정확한 delay)이 NL surface에 entail되지 않으므로, IR은 코드 생성 *이전에* 그 선택들을 승인용으로 *노출*해야 함 — Confirmable이 input ambiguity를 collapse, Executable이 reference trace 생성, Obligation-derivable이 coverage/check 생성. 생성된 idiomatic code(사용자가 못 읽음, temporal 거동 안 보임)가 아니라 IR(읽을 수 있음, IR-SIM으로 temporal 거동 가시화)에서 confirm하는 게 유일하게 합리적인 지점 — 이것이 C1(NL-projectable)·C2(idiom-invariance)가 *requirement*인 이유.
• [CLAIM] Executable + Obligation-derivable은 하나의 IR의 두 *projection*(두 개의 hack 아님) — "왜 simulator AND FSM?"를 선제 차단.

## §5 Timeline IR (the artifact) (flow)
• Design philosophy: idiom-encoded reactive 패턴을 first-class typed field로 대체(wait.edge, wait.for, cycle.count, cycle.until). **[CLAIM — scoped generality]** *8개 typed step operator(+ field-level 확장)*로 우리가 조사·평가한 RA-DSL fragment의 reactive-temporal 구성요소(P1–P14: edge/level/sustain/debounce/hysteresis/periodic/cron/alternation/bounded-repeat/delay-seq/state-var/timer/all-any)를 포섭 — 반면 흔한 consumer RA-DSL은 hysteresis/alternation 같은 패턴을 uniform first-class 구성요소가 아닌 변수·타이머·control-flow idiom으로 인코딩하곤 함. 따라서 IR은 이 구성요소들을 typed field로 *normalize*함(normalized behavioral form, *우리 IR grammar 내에서* canonical — 전역 normal-form 주장 아님) → 여러 valid JoI 인코딩에 걸쳐 confirm된 behavior가 안정적이고, 코드가 못 되는 단일 비교 reference가 될 수 있음.
• [TAB] 8 ops + field extension; trigger-disambiguation rule ("when"→wait edge:none; "whenever/every time"→cycle{wait rising}) — **confirmation *이전*에 적용되는 *default interpretation*(documented prior)으로 frame, 키워드가 intent를 확정한다는 hard claim 아님**(§3.3b); confirmation이 사용자가 override하는 지점. (benchmark에서는 이 동일 convention이 locked *evaluation* label — §9.1.)
• IR→NL projection (confirmation channel): 1:1 deterministic 렌더; 하나의 IR ⇒ 하나의 English. ## §6 IR as Spec: Two Projections (flow)
*(codex #4: §6은 CONCEPTUAL — IR이 두 projection으로 spec 역할을 할 수 있는 semantic design. 여기엔 알고리즘/수치 없음; 그건 §8. §6↔§8 경계 명확히 해 overlap 회피.)*
• §6.1 spec role: 후보 JoI가 *bounded conformance check를 통과하는지* 결정하는 deterministic artifact("옳은지"가 아님 — Rung-1, completeness 아님); "spec" = "무엇이 일어날까"(operational, IR-SIM 경유) + "어떤 transition obligation / scenario boundary가 exercise돼야 하나"(structural, IR-FSM 경유) 둘 다 답함.
• §6.2 **IR-SIM (dynamic projection)** = executable operational semantics → reference timestamped device-service trace; **observation model을 여기서 정의** = (t_ms, service, method, args)의 시퀀스; 왜 formal semantics가 아니라 simulator인가(JoI엔 formal semantics 없음). (정의만; 비교 알고리즘 + tolerance = §8.)
• §6.3 **IR-FSM (static projection)** = 제어흐름 트리에서 derive된 transition obligation(edge/sustain/sequencing/re-arm/register-dep/reachability/first-iter). (어떤 obligation이 존재하나; 어떻게 cover/check하나 = §8.)
• §6.4 왜 JoI에 직접 FSM-FSM이 안 되나(informal semantics, continuous state, idiom multiplicity, bisimulation gap) — SHORT, Related Work로.
• §6.4b **[CLAIM — model checking 아님; formal-verification wedge]** IR-FSM은 *spec의 test-generation/coverage projection이지 model-checking 대상이 아니다*: formal verification은 (i) 형식 모델 AND (ii) expert-authored property를 전제하고 모델이 property를 *모든* behavior에 대해 만족함을 증명함 — 우리 세팅엔 둘 다 없음(oracle 없음; 비개발자 user)이라 우리 기여는 검증의 *상류*(user-confirmed intent에서 reference behavior를 *획득*)에 있고, 검사는 *concrete differential trace testing*(유한 합성 suite 위 IR-SIM vs JoI-SIM)이지 symbolic exhaustive proof가 아님. 따라서 가장 가까운 이웃은 timed-automata model checking이 아니라 **model-based / conformance testing**이며, 그것과도 *intent-derived·user-confirmed* 모델 / construct-derived obligation 열거 / 닫힌 self-correction loop(§8)로 구분됨. **정직한 양보**: formal verification은 우리가 주지 않는 all-input + liveness 보장을 줌; 우리는 exhaustive proof를 spec-acquisition-from-intent + idiom-invariant bounded conformance와 맞바꿈 — *다른 문제*이지 약한 증명이 아님. [전체 대비 §10.3/§10.5]
• §6.5 따라서 verification *principle*: trace(IR-SIM) + obligation 귀속(IR-FSM). **[CLAIM — IR-as-spec hero 아래의 depth]** IR-FSM은 trace oracle을 *보완*함 — transition obligation과 coverage target을 derive해 trace 비교를 ad hoc diff가 아닌 *bounded behavioral conformance*(Rung 1)로 만듦. IR-SIM은 "시도한 입력에서 일치했나", IR-FSM은 "그 입력이 intent를 구분하는 behavior를 cover했나"를 답함. (이건 hero contribution *안의* depth — spec이 hero이고, IR-FSM은 검사를 구조적·진단가능·bounded하게 유지하는 요소이지 제2의 hero가 아님.) *구현된* 메커니즘, claim ladder, coverage generator, mutation 증거, retry는 §8.

## §8 The Verifier (the moat) (flow)
• §8.1 two-layer: L1 static(IR-FSM 위) → L2 trace-equivalence(IR-SIM vs JoI-SIM), short-circuit; tolerance max(500ms,10%). **정직한 scoping (codex)**: L1의 static-analysis *기법*은 신규성 아님 — AWS IoT Events "Run analysis"가 유사한 구조 diagnostics(unreachable-state/missing-trigger) 수행 — 단 L1조차 hand-authored FSM이 아니라 *intent-derived, user-confirmed* IR 위에서 돈다는 점이 다름. hero는 **L2 = derived reference-trace oracle(IR-SIM) + user-confirmed spec 대비 trace-conformance** + confirmation loop. (중요: L1조차 *구조 diagnostics이지 model checking이 아님* — intent-derived IR 위에서 concrete test scenario를 seed/filter할 뿐; behavioral verdict는 L2의 concrete trace-conformance이지 symbolic property 증명이 결코 아님 — §6.4b.)
• §8.2 diagnose + retry: violation → IR-feature-grained hint → sLLM retry (deterministic floor + optional LLM-aided diagnose).
• §8.3 **무엇을 증명하나 — the claim ladder** [CLAIM, 신중 — codex: 변명이 아니라 CONTRACT로 framing]:
  - **Guaranteed**: T1 Timeline-IR determinism(*spec*이 unambiguous; IR-FSM은 그로부터 결정적으로 derive — IR이 spec, FSM은 projection); T2 rejection soundness — flag ⇒ real divergence.
  - **"real divergence"를 정밀 정의(codex 경고, 안 하면 overclaim)**: IR-SIM과 JoI-SIM trace 간 divergence, 단 *기준은* (i) 두 simulator의 operational semantics, (ii) (t,service,method,args) observation model, (iii) max(500ms,10%) timing tolerance, (iv) finite fragment F(guard-discretized alphabet, bounded horizon/register, deterministic). T2 = (i)–(iv) 안에서 reject은 결코 spurious하지 않다. "pass ⇒ correct"는 주장하지 않음.
  - **Covered empirically**: F 위의 Rung-1 bounded transition-boundary coverage.
  - **Exposed**: 모든 flag가 repair를 먹이는 concrete counterexample을 산출.
  - **Measured**: mutation catch rate + survivor 분석 + deployment drift.
  - Abstract/intro 표현 = "bounded behavioral conformance checking under a stated observation model"; "not sound acceptance" caveat는 여기와 §11에만.

## Appendix A — Timeline IR Schema (Reference) *(paper_summary에서 이관; 여기가 canonical)*

### A.1 Top-level
```json
{ "timeline": [ <step>, <step>, ... ] }
```

### A.2 Step grammar (8 ops + field-level extensions)
| Op | 의미 | Key fields |
|---|---|---|
| `start_at` | scenario anchor | `anchor:"now"`, or `anchor:"cron", cron:"<5-field>"` |
| `wait` | block until cond | `cond`, `edge:none\|rising\|falling`, optional `for:"<N> <UNIT>"` |
| `delay` | pause | `duration:"<N> <HOUR\|MIN\|SEC\|MSEC>"` |
| `read` | 값을 local var로 snapshot | `var`, `src:"<Device.attr>"` |
| `call` | device method call | `target:"<Device.method>"`, `args:{...}` |
| `if` | one-shot branch | `cond`, `then:[...]`, `else:[...]` |
| `cycle` | repeat body | `period:"<N> <UNIT>"`, optional `until:"<expr>"`, optional `count:"<var>"`, `body:[...]` |
| `break` | exit nearest cycle | — |

Field-level 확장(8-op count와 직교): `wait.for`(sustained cond), `cycle.count`(tick-index int; lowering = `n:=0` + body + `n=n+1`).

### A.3 Expression grammar
- **Literals**: numbers, strings, booleans.
- **Device attribute reference**: `Device_id.attr` (예: `TempSensor_1.temperature`).
- **Local variable reference**: `$varname` (prior `read`) 또는 count var bare name (예: `n`).
- **Clock**: `clock.time`(4-digit `hhmm` int), `clock.date`(`YYYYMMdd`), `clock.dayOfWeek`(`"MON".."SUN"`).
- **Operators**: `+ - * / %`, `( )`, `== != < > <= >=`, `&&`, `||`, `!`.
- **Convention β**: `args` string은 `.`/`$`/연산자 포함 시 expression, 아니면 literal.

### A.4 Trigger mapping (confirmation 이전 default interpretation; locked *evaluation* convention)
> 이들은 generator의 **confirmation 이전 default reading** — documented prior이지 키워드가 intent를 유일하게 확정한다는 주장 아님(§3.3b). deployment에선 사용자가 IR confirmation으로 각 행을 override함. benchmark에선 이 동일 표가 accuracy를 재현 가능하게 만드는 *locked evaluation label*(§9.1).

| English | IR pattern |
|---|---|
| `if X, do Y` | `if` one-shot (no wait, no cycle) |
| `when X, do Y` | `wait(edge:"none") + Y` — one-shot level wait |
| `whenever X, do Y` / `every time X, do Y` | `cycle { wait(edge:"rising"); Y }` |
| `X for at least N` / `X for N` | `wait(edge:"none", for:"N UNIT")` by default |
| `every N UNIT, do Y` | top-level `cycle(period:"N UNIT") { Y }` (not cron) |
| `every N UNIT, alternate A and B` | `cycle(period:"N UNIT", count:"n") { if(n%2==0){A} else {B} }` |
| `every N UNIT, do X; stop after K times` | `cycle(period:"N UNIT", until:"n >= K", count:"n") { X }` |

### A.5 Validator rules
- Top-level `timeline`(non-empty list); 첫 step은 `start_at`.
- `cycle.period` 필수(authoritative polling cadence); `cycle.body`에 ≥1 `delay` 또는 edge-`wait`(cadence guarantee); `cycle.count`(있으면) non-empty string id; nested cycle 금지.
- `wait.edge` ∈ `{none, rising, falling}`; `wait.for`는 `delay` duration grammar.
- 모든 `Device.attr`/`Device.method`는 service catalog에 존재해야.

### A.6 Reject conditions
- period/interval 없는 cycle; 미정의 device/attribute 참조; nested loop 요청.
