### AutoTap (ICSE 2019) [full-text]
- **What/relation:** Non-experts click *negative LTL invariant properties* ("X & Y should never co-occur"); system synthesizes/repairs TAP rules to satisfy them. Closest "non-expert pins a spec" neighbor — but it's property-authoring (the exact AutoTap-style misread OVLA must avoid), and it *generates to satisfy*, not *verifies generated code against intent*.
- **Flow:** Hook = the gap (novices write buggy TAP for complex/temporal behaviors) with a running example → names 3 correctness goals **property-compliant / accommodating / valid** in §1 → **formative User Study 1 derives the design** → interface → synthesis → eval = **summative User Study 2 + synthesis benchmark** → threats → related work. Reusable: named-goals-in-§1; formative-then-summative studies.
- **User study:** TWO controlled MTurk studies. **Study1 (formative) N=71**: free-text 10 properties → 7 templates (κ=0.62). **Study2 (summative) N=78, between-subjects Rules vs Properties**, 7 of 14 tasks; metrics = correctness 0/partial/complete (2 coders), confidence 5-pt, **SUS**; stats = **Holm χ² per task** (sig on 5/14, favoring Properties), Mann-Whitney U for SUS (n.s.). Plus: synthesized 157/158, fixed 23/25 mutants. Honestly concedes tasks over-represent property-expressible behaviors.
- **Vocab:** "property-compliant / satisfy / model checking / violation"; reference = user-specified **LTL property**; "execution", "transition system", "behavior"; "TAP rule / TAP program"; "desired property / user-specified properties / express intent"; "bug / mistake / violation / mutation"; users **specify via mouse-clicks**.
- **Takeaway:** (1) The **two-study (formative+summative between-subjects, correctness-coded, SUS, χ²) pattern is the gold standard** — model RQ1's summative arm on it. (2) Differentiate hard: AutoTap = user *authors* negative properties; OVLA = user *confirms* a positive NL rendering. (3) Reuse its honest task-representativeness caveat.

### Helping Users Debug TAP (IMWUT 2022) [full-text] — **most reusable for RQ1**  → **전체 deep-dive = `tapdebug_deepdive.md`** (32p 정독: 무프라이어워크 정당화 / 풀 procedure / 모든 실험·통계 / threats / RQ1 borrow list)
- **What/relation:** First empirical study of *end-to-end TAP debugging* + two tools (Explicit/Implicit feedback) that SAT-repair human-authored TAP from user feedback on simulated traces. Non-experts debugging buggy automations in a 3-D sim — directly the RQ1 shape.
- **Flow:** Hook = **missing-study gap** ("no prior work studied end-to-end debugging") → 4-stage cognitive model (**Misbehavior Identification → Fault Localization → Patch Creation → Patch Refinement**) that *organizes the whole paper* → explicitly **justifies the simulator** (cost/privacy/time) → tools → SAT synthesis → methodology → results-by-stage obstacle taxonomy. Reusable: cognitive-stage scaffold doubling as results structure; sim-justification verbatim-usable.
- **User study:** Controlled remote, **between-subjects, 3 conditions (Control / Explicit-FB / Implicit-FB), N=30 (10 each), Prolific US, +45 pilots**. 3-D sim "Home I/O". 5 debugging tasks (nearly-correct-but-buggy, known ideal fix), spanning **Over-/Under-Automation × discovered-immediately/later**. Metrics: correctness **0-3 coded by 2 raters, Cohen's κ=0.86**; **16-obstacle taxonomy**; SUS; time; clarification count. Stats: **Kruskal-Wallis (BH-adjusted) + Mann-Whitney U**. Proved: without tools users mostly fail (0-2/10 fully correct); tools significantly help and **shift role from patch *creator* to *reviewer***; obstacles only *alleviated, not eliminated* (10/100 sessions rejected all correct patches).
- **Killer finding for OVLA:** **Intra-rule-Logic obstacle — users misread IF(event) as a checked state / WHILE(state) as an event — 21/50 Control sessions, ALL 21 failed.** Direct proof that reading raw rules/code does NOT convey reactive-temporal intent to non-experts. Caution: **Misreading of machine-generated text *increased* with tools (1 vs 28/18)** → the IR rendering itself can be misread; RQ1 must measure it.
- **Vocab:** "debugging / fix / patch"; reference = "intended behavior / ideal modification", user "feedback on what should/shouldn't have happened"; **Over-Automation = false-positive, Under-Automation = false-negative**; "trace = history of automated+non-automated events"; "misbehavior"; "TAP rule / TAPs / patch"; "bug / fault / obstacle / mutation".
- **Takeaway:** (1) **Use this as the RQ1 template wholesale** — 3 conditions = IR-rendering/code/trace, ~5-6 faulty-but-near-correct tasks, sim, 0-3 coding with κ, obstacle taxonomy, SUS+time, KW+MWU, pilots, randomized order. (2) Its Intra-rule-Logic result is OVLA's empirical justification for the NL IR rendering; its Misreading result is the risk to measure. (3) Adopt **Over-/Under-Automation** and the **4-stage model** as standard scaffolding.

### ChatIoT (IMWUT 2024) [full-text]
- **What/relation:** Cloud-LLM (GPT-3.5) NL→Home-Assistant TAP via compressive prompting + **4-agent pipeline (Preprocessor/Service Creator/TAP Generator/Evaluator)** + DRL edge model customization. **Its LLM "Evaluator" = the cloud-LLM-judge baseline OVLA argues against.**
- **Flow:** Hook = capability/convenience ("prior approaches limited, not zero-code") → 2 challenges→2 contributions → related work with **Table-1 capability matrix (learning-cost × interaction × customization)** → system → eval (accuracy + token headline vs Zero-shot/CoT/ReAct → **ablation isolating Evaluator = +3-5pp** → case study + thin user study LAST).
- **User study:** Case study (1 DIY home, end-to-end walkthrough, ~40s/$0.0028) + **thin user study: 15 users → 60 requests → 91.57% accuracy vs hand-made GT. No conditions, no SUS, no coding, no stats.** Measured generation accuracy, never user-side verifiability.
- **Vocab:** **"Evaluator"** (LLM agent; "evaluates format compliance and request satisfaction", "can correct format discrepancies/irregular property values") = the cloud LLM-judge; GT = "manually crafted TAP / benchmark"; "TAP / IF…WHILE…THEN"; "user request/requirement/intention"; "error/hallucination/format discrepancy"; **"edge" = orchestrator only; the LLM is cloud**; "zero-code generation"; "AskUser".
- **Takeaway:** (1) **ChatIoT's Evaluator is the concrete LLM-judge baseline** — cite precisely: it's an LLM, catches only format/value errors, cloud-based, +3-5pp (Table 6). (2) **Steal the Table-1 capability-matrix shape** for RQ5. (3) Its "edge" isn't on-device LLM → OVLA's no-cloud is a real differentiator; don't imitate its thin accuracy-only "user study."

### SimuHome (ICLR 2026) [full-text — 10pp main body; appendices not read]
- **What/relation:** Matter-grounded, **time-accelerated deterministic smart-home simulator** + **600-episode benchmark**, evaluating **18 LLMs**. **Premier OVLA-motivation citation:** workflow scheduling is the hardest category and persists; self-correction fails.
- **Flow:** Hook = capability-gap escalation (explicit→implicit→dependency→**temporal/workflow**) → gap (existing benchmarks static, no continuous dynamics/dependencies/scheduling) → 2 contributions (simulator + benchmark) → related work differentiates from HomeBench/Sasha/SAGE → eval (18 models × 6 query-types × feasible/infeasible → error analysis → role of tool feedback → disentangle framework vs model). Reusable: **two-tier eval** (deterministic state-check oracle + LLM-judge with κ=0.826 vs humans); **feasible+infeasible task pairs**; error taxonomy.
- **User study:** **NONE.** Validated by: benchmark annotator κ=0.92; LLM-judge κ=0.826 vs humans (3× majority); 18-model success-rate/time/error eval. **Key findings:** workflow scheduling hardest across all 18; reasoning models (GPT-5.1) help but **3-5× slower → impractical real-time** (>100s); **"agents cannot detect their own mistakes"; self-correction recovery ≤18.5% (QT4-2)/0% (QT4-3); even oracle-fed recovery only 55-67%**; fine-tuning didn't fix feasible scheduling.
- **Vocab:** "verifiable/structured **goal**" (= oracle, but benchmark-authored); the sim **deliberately does not validate at schedule time** ("no error returned"); "deterministic state transitions (tick-based)"; "environmental variables"; "**workflow** / command / tool call / Matter cluster-attribute-command"; error taxonomy (Environment Perception/Intent Inference/Device Control/Action Planning/**Temporal Reasoning**; Contradiction Mishandling/Blindness); "real-time/near-instant latency".
- **Takeaway:** (1) **The motivation backbone** — 18 LLMs fail workflow scheduling, reasoning too slow, **self-check/post-hoc does NOT work** → pre-deploy verification (not self-check) is needed. Update catalog's stale "18 agents" note (confirmed: 18 models, GPT-5.1, 600 episodes, ICLR'26). (2) Reuse **feasible+infeasible pairing + two-tier eval + error taxonomy**. (3) Borrow "**structured goal**" for the oracle; differentiate: theirs is benchmark-authored & not validated at schedule time; OVLA's is **user-confirmed & validated pre-deploy**.

### TaskSense (SenSys 2025) [full-text]
- **What/relation:** Cloud-LLM NL→**Sensor Language** (tools=vocab, dependencies=grammar)→tool-call **plans** for complex sensor **QA**. Closest "grammar/FSM + NL re-translation" neighbor; its checks + re-translation are what OVLA differentiates from.
- **Flow:** Hook = LLMs lack sensor capability/dependency knowledge → mis-judge solvability/wrong plans; "translation" metaphor → **measurement-study motivation** (quantifies HuggingGPT failure distribution on 38 queries) → design (sensor language + **solvability check + grammar/DAG-subgraph check** + dynamic adaptation + response gen) → eval (plan/exec/response accuracy vs HuggingGPT/Sasha, 6 LLMs; ablation; overhead; end-to-end app). Reusable: **measurement-motivation**; deterministic structural checks separated from LLM; 3-tier accuracy decomposition.
- **User study:** **NONE.** Validated by accuracy metrics (Planning/Execution/Response, GT = manual plans), microbenchmarks, ablation, Jetson Orin case study.
- **Vocab:** **"solvability check / grammar check (plan-DAG ⊆ grammar-DAG) / plan verification"** (= structural well-formedness ≈ OVLA L1, explicitly NOT intent-conformance); "ground-truth plan"; "execution / plan execution results"; "plan / tool call / sensor language / data & control dependency / DAG"; "user query/intention"; "misjudging solvability / wrong plan / hallucinated tools"; **"response generation = translate output back to NL" = an inspectable answer, NOT a code round-trip**; Jetson Orin but LLM is cloud.
- **Takeaway:** (1) Its two checks = **OVLA's L1 analogue** — cite to show structural/grammar checks are known/deterministic but verify only well-formedness, so **L1 is not the novel part (L2 + confirmation is)**. (2) Adopt the **measurement-motivation move**. (3) Re-translation works for QA (inspectable answer); reactive automation's "output" is future device actions → can't rely on LLM answering back → deterministic behavioral verifier needed. No human study → reinforces the gap.

### GPIoT (SenSys 2025) [full-text]
- **What/relation:** Fine-tunes **local SLMs (Llama2-13B INT8 + LoRA "PECT")** in a 3-SLM pipeline (decompose→requirement-transform→codegen) for **IoT algorithmic code** (ECG/HAR). Shares OVLA's on-device/no-cloud premise (baseline-axis sibling) but targets code-with-executable-oracles, not reactive automation, with **no intent verification**.
- **Flow:** Hook = **cloud-RAG is privacy/cost/instability-bad → local SLM + fine-tune** (≈ OVLA's no-cloud motivation, pre-made) → **measurement-motivation** (§2: 28%/53.4% failure stats) → system → eval (per-app metrics vs 7 baselines → IoTBench breakdown → ablations → **§6.6 user study LAST**).

- **Vocab:** "correctness / code review / pass@k / FCR"; oracle exists = "test cases / detection accuracy / reference / ground truth"; "execute the code / inference time"; "IoT program / task / sub-task / **task specification** / decomposed task"; "user requirement / problem statement → spec → code"; "bug / hallucination / format incompatibility / domain misalignment"; **"locally deployable SLM / privacy-preserving / commodity GPU / edge server"**; "program synthesis / code generation"; pass@1/5, URC, STC, BLEU.
- **Takeaway:** (1) **Pre-writes OVLA's no-cloud motivation** + is the **local-SLM baseline sibling** (differentiator: OVLA uses ≤9B 4-bit *without* fine-tuning). (2) **Cleanest "our domain differs on the oracle" cite** — GPIoT can use pass@k/SonarQube because algorithmic code has an executable oracle; reactive automation doesn't → OVLA *derives* its oracle from the confirmed IR. (3) Adopt **"task specification"** and **"locally deployable SLM / privacy-preserving"**. Its 20-user no-stats Likert is the *minimum* a SenSys reviewer accepts; RQ1 should exceed it (per TAP-Debug/AutoTap rigor).

### FLOW PATTERNS
- **Templates:** (1) capability-gap escalation hook (all 4 LLM-IoT papers + AutoTap; all lead with capability, not safety); (2) missing-study gap hook (TAP-Debug); (3) **measurement-motivation** (TaskSense §3, GPIoT §2 — quantify baseline failures to justify each component); (4) named-goals/contributions in §1 (AutoTap); (5) **capability-comparison matrix in related work** (ChatIoT Table 1, Survey Table I); (6) eval arc = setup→headline-vs-baselines→ablation→overhead→**user/case study LAST** (LLM-IoT papers), whereas HCI/AutoTap put formative study first.
- **Fit for OVLA:** fuse **missing-mechanism hook + measurement-motivation**, lead with the **safety gap (silent-wrong)** — the deliberate inversion of the 4 capability-first hooks. ### GAPS / OPENINGS
1. **No credible controlled non-expert *verification* user study exists** in the NL→automation/IoT-code corpus. TaskSense/SimuHome = none; ChatIoT/GPIoT measure generation accuracy / preference; the only rigorous human-debugging study (TAP-Debug) is on *human-authored TAP repair*, not LLM-generated-automation verification. → OVLA's RQ1 is genuinely novel **if** executed at the TAP-Debug rigor bar.
2. **Self-check / LLM-judge demonstrably fails on reactive-temporal correctness** (SimuHome self-correction ≤18.5%/0%, oracle-fed ≤67%, "cannot detect their own mistakes"; ChatIoT Evaluator catches only format errors, +3-5pp). → A deterministic LLM-free pre-deploy behavioral verifier is an unfilled slot.
3. **No on-device-LLM + verified pipeline.** All generators here are cloud (ChatIoT, TaskSense); GPIoT is local but fine-tunes, targets algorithmic code with an oracle, and verifies no intent. → OVLA's ≤9B / no-fine-tune / no-cloud / verified combo is empty.
4. **No positive-behavioral, user-confirmed oracle for reactive automation.** AutoTap = negative LTL invariants; SimuHome = benchmark goals (not validated at schedule time); TaskSense = plan well-formedness; ChatIoT = author-made GT. → OVLA's user-confirmed positive behavioral spec (Timeline IR) as the deploy-time reference oracle is the unfilled wedge (corroborates the catalog's "T6 empty").
5. **"Event vs state" misreading empirically defeats non-experts reading raw rules** (TAP-Debug 21/21 Control failures). → OVLA's NL IR rendering has a measured reason to exist; the open question RQ1 answers is whether the rendering fixes it *without* its own misreading (TAP-Debug's Misreading caution).

## Phase 2a — web SenSys / IoT (LLM-generated automation, on-device, verification)

> Web sweep, SenSys 2022+ and adjacent venues (SenSys/MobiSys/IMWUT/ACL benchmarks). Bodies read where openly readable (arXiv HTML). Full-text vs abstract-only marked honestly. Does NOT re-cover the 7 Phase-1 papers. Produced by `related-works-analyst`.

### AutoIoT: Automated IoT Platform Using LLMs (arXiv 2024 / preprint) [full-text]
- **What/relation:** LLM (cloud GPT) end-to-end: photo+manuals of devices -> extract device info -> generate trigger-action rules -> **generate formal Maude code via a "code adapter"** -> **formal verification (Maude model checking / state search) for conflict detection & avoidance**. Closest "LLM-generated automation + formal check before deploy" neighbor. BUT the check is **rule-vs-rule conflict** (state/environment/cascading), NOT rule-vs-user-intent; no user confirmation; verifies safety properties, not "did we build what you asked." This is the AutoTap/TAPInspector lineage with an LLM front-end.
- **PAPER FLOW:** Hook = usability barrier (non-experts can't author rules) + safety (rule conflicts) -> **Table I capability matrix** dismissing commercial platforms (HomeKit/SmartThings/Nest = no auto-generation, no auto-conflict-code) and academic (IoTMediator/CP-IoT = detect conflicts but require manual code) -> preliminaries on LLMs + formal verification -> 2-component system (Rule Generation; Conflict Detection & Avoidance) -> **"code adapter" separates logical reasoning from target-language syntax** (so LLM emits Python-function templates, adapter lowers to Maude) -> eval = pipeline time-cost by device-count & LLM choice -> case study (10 real Mi Home devices, dual deploy app + Raspberry-Pi Python). Headline = "no conflicts among generated rules." Reusable moves: Table-I capability matrix; **"separate logic from target-DSL syntax" = exactly OVLA's IR-as-generation-template (IR's 3rd role)**; formalize fault types up front.
- **USER/CASE STUDY:** NONE (no participants). Validated by a **single self-run 10-device case study** + per-stage latency. No baseline, no ablation, no stats. Weak validation bar (good contrast: OVLA's RQ1 must clear far higher).

### Alive2: Bounded Translation Validation for LLVM (PLDI 2021) [full-text via authors' summary + canonical secondary; eval numbers second-hand-confirmed]
- **What/relation:** A *translation validator*: given an LLVM IR function before and after an optimization pass, it checks (via SMT/Z3) that the optimized version **refines** the original, fully automatically, no annotations, no changes to LLVM. This is OVLA's exact shape — **a deterministic, automatic checker comparing a transformed artifact against a reference under a precise semantic relation, designed to flag only real divergences** — except OVLA's relation is trace-equivalence-within-tolerance over IR-sim vs code-sim, and OVLA's reference is a user-confirmed IR rather than the pre-optimization IR.
- **PAPER FLOW:** Problem = compiler optimizations silently miscompile; prior verified-compiler / Alive1 work needs annotations or hand-written patterns → contribution = automatic, UB-aware, bounded validator → semantics of LLVM IR incl. undefined behavior/poison → encoding to SMT → **eval = run over LLVM's own unit-test suite + measure bugs found, false alarms, solver cost.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the validator is good:** (a) **real bugs surfaced** — ran Alive2 over LLVM's existing optimization unit-test suite and **found 47 new bugs, 28 fixed**, plus **8 patches to the LLVM Language Reference** (the semantics doc itself) — i.e. the validator's value is shown by *finding genuine divergences a mature, heavily-tested system missed*; (b) **false-alarm discipline** — explicitly engineered to **avoid false alarms**, and the headline argument is that UB handling exists *precisely because* mishandling UB "would make the number of false alarms... impractical"; (c) **solver cost / applicability** — fraction of functions validated within timeout, where it times out.
  - **Dataset:** LLVM's own unit tests (the optimizer's regression corpus) — an *independent, pre-existing* corpus they did not author for the checker.
  - **Baselines:** prior Alive (pattern-limited, no loops/UB), and implicitly "LLVM's existing test suite + human review," which the 47 new bugs beat.
  - **Metrics:** bugs found / fixed; upstreamed semantics patches; false-alarm avoidance (qualitative-by-construction + reported low); validated-vs-timeout rate; per-pass coverage.
  - **HOW THEY DEFUSE CIRCULARITY / "checker defines correctness":** three moves. (1) **Soundness is one-directional and named "bounded":** a *pass* is not a proof of full correctness — they openly state that bounding loop unrolling means **"there are circumstances in which it misses bugs"** (incompleteness conceded up front); the strong claim is only on the *flag* side (a reported refinement violation is real). (2) **The reference semantics is external and was itself stress-tested** — running the validator drove **8 corrections to the official LLVM Language Reference**, i.e. they treat "the semantics the checker assumes" as a separately-auditable, fixable object rather than ground truth by fiat. (3) **Independent corpus** — bugs were found in LLVM's *own* tests, not tests written to make the checker look good.
  - **"No false positives by construction"-style argument:** yes — refinement is defined so that *any* counterexample is a concrete input on which target misbehaves where source did not; UB/poison are modeled so legal optimizations are not mis-flagged. The whole UB-modeling effort is justified as **false-alarm suppression**, not completeness.
- **VOCABULARY:** *translation validation*; *refinement* (target refines source; not symmetric equivalence — the load-bearing relation); *undefined behavior / poison / immediate-UB* (the false-alarm sources that must be modeled); *source vs target*; *bounded* (loops unrolled to a bound → incomplete); *sound* (used for the flag direction); *counterexample* (concrete witnessing input); *fully automatic*; *miscompilation*.
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Adopt the asymmetric "rejection-sound, bounded-incomplete" framing verbatim** — OVLA already claims rejection soundness + not-model-checking; Alive2 is the canonical precedent for stating "a flag is a real divergence; a pass is bounded, not a proof" and being *rewarded* for the honesty. Cite it for the claim shape.
  2. **Make false-alarm avoidance a first-class, separately-reported result** (Alive2's central engineering story is "we model UB *so we don't false-alarm*"). OVLA's analogue = the ±500 ms tolerance + fail-closed + idiom-invariance; report **FP rate / no-false-alarm separately from kill rate**, and explain *what was modeled to avoid FPs* (tolerance, idiom unrolling) the way Alive2 explains UB.
  3. **Find real bugs in an artifact you did NOT curate for the verifier** — Alive2's strongest evidence is 47 bugs in LLVM's own suite. OVLA's RQ3 analogue: run the verifier on **LLM-generated programs nobody wrote to be caught** and report genuine silent-misbehavior catches (the 16%→0% headline), not just synthetic mutants.
  4. **Treat the assumed semantics as auditable** — Alive2 upstreamed 8 LangRef fixes. OVLA should explicitly say the IR/JoI semantics the simulators encode is a *fixable, audited* object (answers "who validates the validator": you patch the semantics when it's wrong, as Alive2 did).

### Semantic Evaluation for Text-to-SQL with Distilled Test Suites (EMNLP 2020) [full-text via authors' abstract/eval summary + canonical secondary]
- **What/relation:** Replaces brittle string-match SQL evaluation with **execution-based semantic equivalence**: a predicted query is "correct" iff it returns the same denotation as the gold query **on a distilled suite of databases chosen to expose any behavioral difference.** This is *exactly* OVLA's "compare the generated artifact to a reference by running both and diffing observable results," and its **neighbor-query construction is a mutation-adequacy procedure** — the closest methodological sibling to OVLA's construct-derived boundary-event synthesis.
- **PAPER FLOW:** Problem = exact-set-match has false negatives (semantically-equal SQL written differently) and single-DB execution has false positives (accidental agreement on one DB) → contribution = **test-suite accuracy**, a tight, efficient upper bound on true semantic accuracy → method (neighbor queries + coverage-guided fuzzing + distillation) → **eval = measure the metric's own FP/FN against human-judged semantic correctness, then re-score 21 leaderboard models.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the *oracle* is good (this is the key inversion — the experiment validates the checker itself):** the metric's **false-negative rate vs true semantic accuracy = 2.5% average, 8.1% worst-case** for the old Spider metric, driven down by the distilled suite; and elimination of single-DB false positives. They literally **quantify the oracle's own error rates** rather than assuming it correct.
  - **Mutation-style adequacy = "neighbor queries":** they **systematically modify the gold query** (swap comparators, columns, aggregates, join conditions, etc.) to produce *near-miss* queries, then **distill databases that distinguish the gold from its neighbors** — a database is kept only if it makes gold and a neighbor return different denotations. This is OVLA's pattern precisely: *a mutant the test suite must kill* ⇒ *a test (DB) that separates correct from mutant*. **Code coverage of the gold query** is the adequacy target the fuzzed DBs must hit.
  - **Distillation procedure:** generate many random DBs by fuzzing, keep a minimal subset that preserves all the discriminations (high coverage, small suite) → cheap to run at eval time.
  - **Dataset / scale:** distilled suites for **11 text-to-SQL datasets** (Spider, SParC, CoSQL, Academic, ATIS, GeoQuery, IMDB, etc.); re-scored **21 Spider-leaderboard models.**
  - **Baselines:** exact-set-match (the incumbent, shown to false-negative) and naive single-DB execution accuracy (shown to false-positive).
  - **HOW THEY DEFUSE CIRCULARITY / "the metric defines correctness":** (1) **explicit upper-bound framing** — test-suite accuracy is proven/argued to be a **tight upper bound on true semantic accuracy**, so it can only over-credit, and the gap is bounded and measured (2.5%); they never claim it equals truth. (2) **Human adjudication of the metric** — **manually verified the metric is correct on 100 examples** (an *independent human* ground truth used to validate the *automatic* oracle). (3) **Adopted as the official metric** by Spider/SParC/CoSQL — external community validation. (4) The neighbor-query set makes the discrimination target *explicit and inspectable* rather than implicit.
  - **"No-false-X by construction":** the suite is *constructed* to kill every neighbor it can, so a passing query is indistinguishable-from-gold across all distinguishing DBs found → false positives bounded by coverage of the neighbor set; FN is what they measure and minimize.
- **VOCABULARY:** *semantic accuracy* (the truth target) vs *denotation accuracy* (execution result equality) vs *execution accuracy* (single DB) vs *exact-set-match* (string); *test-suite accuracy* (their metric); *neighbor queries* (the mutants); *distillation* / *code coverage*; *tight upper bound*; *false positive / false negative* (of the metric itself).
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Quantify the verifier's own FP/FN against an independent human-judged set, and report it as a *property of the oracle*, exactly as they report the metric's 2.5% FN** — this is the single most direct rebuttal to "your verifier defines correctness." OVLA's RQ2 should include a *human-adjudicated* "is this flagged divergence a real intent fault?" pass over a sample (their "100 examples manually verified" move).
  2. **Reframe construct-derived event synthesis as "neighbor-query distillation"** — present mutants as *neighbors of the confirmed IR* and boundary events as *the tests distilled to separate the IR from its neighbors*. This gives OVLA's mutation methodology a peer-reviewed name and a coverage-adequacy story (kill every neighbor the suite can distinguish).
  3. **Use the "tight upper/lower bound, not truth" hedge** — OVLA can say "verifier-pass is an *upper bound* on conformance under the synthesized suite; the residual gap is measured (survivor characterization + mutation Ĉ)," paralleling their bounded-FN argument. Avoids over-claiming while still being quantitative.
  4. **Lean on external adoption / independent corpora** — they became the official Spider metric; OVLA's analogue is using gt_ir labels produced *independently* of the verifier's simulator where possible, and disclosing where they share the simulator (the known circularity caveat) the way this paper discloses its bounded FN.

### Csmith — Finding and Understanding Bugs in C Compilers (PLDI 2011) [full-text via authors' preprint summary + canonical secondary]
- **What/relation:** Randomized **differential testing** of C compilers: generate random, **UB-free** C programs, compile with multiple compilers/versions, run, and flag any **output divergence**; a **majority vote** across implementations picks which compiler is wrong. Mirrors OVLA's "run two semantically-independent executions of the same intent and diff observable behavior, fail on divergence" — and its **two defenses against false alarms** (UB-free generation + voting oracle) are the template for OVLA's "a flag is a *real* divergence" claim **without** a trusted golden reference.
- **PAPER FLOW:** Problem = compilers silently miscompile valid code, hard to test because no oracle → contribution = Csmith (random generator that *avoids* UB so divergence is unambiguous) + differential-testing harness + 3-year bug-hunt → eval = **bug counts by type, per-compiler, plus throughput / triage methodology.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the approach works:** **raw genuine-bug yield** — **>325 previously-unknown bugs over 3 years (79 GCC + 202 LLVM reported), most fixed**, and the killer qualitative result: **"every compiler we tested was found to crash and to *silently generate wrong code* when presented with valid input."** That "silently wrong on valid input" framing is *verbatim* OVLA's RQ3 thesis.
  - **Bug taxonomy (directly borrowable for RQ3):** **crash bugs** (compiler aborts) vs **wrong-code bugs** (runtime produces wrong results) vs **silent wrong-code bugs** (wrong code with *no warning*) — OVLA's "silent misbehavior / silent-wrong" maps onto the third class exactly.
  - **Dataset:** millions of *machine-generated* random programs (not a fixed corpus) — adequacy comes from generator coverage of the C feature space, reported as feature/throughput stats.
  - **Baselines / comparison:** other random generators (shown to generate UB ⇒ false alarms); the implicit baseline is the compilers' own enormous test suites, which missed all 325.
  - **Metrics:** bug count by compiler/type; bug-finding throughput; test-case-reduction effort (they automate shrinking divergent programs to minimal reproducers); breadth ("every compiler").
  - **HOW THEY DEFUSE CIRCULARITY / "no trusted reference":** this is Csmith's signature and the most relevant move for OVLA. (1) **No single golden compiler is trusted — a *majority vote* across multiple independent compilers is the oracle**, so "wrong" = "disagrees with the consensus," removing reliance on any one reference defining correctness. (2) **UB-free generation eliminates the dominant false-alarm source by construction** — if a program had undefined behavior, divergence would be *legal* and a false alarm; by emitting only UB-free programs, *any* observed divergence is necessarily a real miscompilation. (3) **Every flag was confirmed by developers fixing it** — external human/maintainer adjudication of each reported divergence as a genuine bug.
  - **"No-false-alarm by construction":** explicit — UB-freedom is the construction-time invariant that guarantees a divergence is a real bug, not differing-but-legal behavior.
- **VOCABULARY:** *differential testing*; *voting / majority oracle*; *undefined / unspecified behavior* (the false-alarm source eliminated by construction); *miscompilation*; *wrong-code bug* vs *crash bug* vs ***silent wrong-code bug***; *test-case reduction*; *valid input* (UB-free); *divergence / inconsistent behavior*.
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Adopt the bug taxonomy and the "silently wrong on valid input" framing for RQ3** — OVLA's headline (silent-wrong 16%→0%) is the *exact* phenomenon Csmith made canonical; cite it and name OVLA's caught class "silent wrong-behavior" in their lineage.
  2. **The voting/majority-oracle is OVLA's circularity escape hatch** — where OVLA can run a *third, independently-built* execution (e.g. a reference interpreter not sharing the verifier's simulator, or multiple-ground-truth IRs / multi-GT) and flag only on consensus divergence, it directly answers "your simulator defines correctness." OVLA's existing "multi-GT verifier" idea = a voting oracle; frame it that way and cite Csmith.
  3. **"Eliminate the false-alarm source by construction" is the model for OVLA's tolerance + idiom-invariance** — Csmith removes UB so divergence is unambiguous; OVLA removes timing jitter (±500 ms) and idiom-encoding differences so a trace divergence is unambiguously an *intent* fault. State it as a construction-time invariant, not a heuristic.
  4. **Yield-on-uncurated-input is the strongest RQ3 evidence** — Csmith's 325 bugs in mature compilers >> any synthetic kill rate. OVLA's analogue: report genuine silent faults caught in *naturally-generated* LLM output (not just mutants), and have an external/human pass confirm each flagged divergence is real (their "developer fixed it" = OVLA's human-adjudicated TP confirmation).

## PHASE-2b SYNTHESIS — menu of credible verification-experiment designs

**The three papers form a coherent menu for OVLA's verifier-evaluation, all sharing OVLA's "compare a generated artifact to a reference by execution, flag divergences, claim the flag (not the pass) is sound" shape:**

| Design move | Alive2 | Test-Suite Acc. | Csmith | OVLA mapping |
|---|---|---|---|---|
| **Genuine bugs in an *uncurated* artifact** | 47 in LLVM's own tests | — | 325 in mature compilers | **RQ3:** catch silent faults in *naturally-generated* LLM programs, not only mutants |
| **Mutation/neighbor adequacy** | (UB cases) | **neighbor queries + coverage-distilled DBs** | feature-space coverage | **RQ2:** reframe construct-derived synthesis as "neighbor-of-IR distillation"; kill rate = neighbor-kill |
| **Quantify the *oracle's own* FP/FN vs human truth** | low-FP by construction | **2.5% FN measured; 100 examples human-verified** | dev-confirmed each flag | **RQ2:** human-adjudicate a sample of flags; report FP/FN *of the verifier* separately |
| **Asymmetric soundness ("flag real, pass bounded")** | **explicit "bounded, misses bugs"** | "tight upper bound, not truth" | — | **claim shape:** rejection-sound + not-model-checking, stated as a feature |
| **Independent / non-trusted reference** | LangRef audited & patched | official-metric adoption | **majority-vote oracle** | **circularity:** multi-GT = voting oracle; independent reference-sim; audit shared-sim caveat |
| **False-alarm source removed by *construction*** | **model UB** | suite kills neighbors | **UB-free generation** | tolerance (±500ms) + idiom-invariance as construction-time FP-suppression invariants |

### How the strongest of them handle OVLA's exact circularity / soundness problem
OVLA's vulnerability: *"the verifier and the eval labels share the same simulator, so soundness is by-construction relative to the IR."* The three papers give three layered, citable defenses, in increasing strength:

1. **Disclose + bound (Alive2, Test-Suite Acc.).** Both *concede incompleteness openly* and turn it into a measured quantity: Alive2 "misses bugs" under its loop bound; Test-Suite Acc. has a measured 2.5% residual FN as a "tight upper bound." **OVLA move:** state plainly that pass = bounded conformance under the synthesized suite (not proof), and *quantify the residual* (survivor characterization, Good-Turing Ĉ) — honesty is rewarded, not penalized, at top venues.

2. **Human-adjudicate the oracle on an independent sample (Test-Suite Acc.).** Their "100 examples manually verified" + Csmith's "developers fixed every reported bug" both use *external human judgment* to confirm the automatic oracle's verdicts. **OVLA move (highest-value, cheap):** take a sample of verifier *flags* (and a sample of *passes*), have humans/an independent gt judge whether each is a real intent divergence, and report verifier-FP/verifier-FN as a measured property — this is the most direct rebuttal to "the checker defines correctness."

3. **Use a non-shared / voting reference where possible (Csmith, Alive2).** Csmith's **majority-vote across independent compilers** and Alive2's **externally-audited LangRef** both avoid any single self-authored reference being correct-by-fiat. **OVLA move:** the **multi-GT verifier** is a voting oracle — frame it explicitly as such and cite Csmith; and where feasible, build at least one reference path (e.g. a second IR-interpreter, or gt_ir labels) that does *not* reuse the code-sim, so the flag rests on *agreement of independent executions*, not one simulator's say-so.

**Net:** RQ2 should report **(kill rate on neighbor-mutants) + (verifier FP/FN vs human-adjudicated sample) + (residual-fault bound)** as three separate numbers — the Test-Suite-Accuracy structure. RQ3 should report **genuine silent-wrong catches in uncurated LLM output**, taxonomized as Csmith's *silent wrong-behavior*, with each flag externally confirmed — the Csmith/Alive2 "real bugs in real artifacts" structure. The soundness claim should be stated Alive2-style: **rejection-sound, bounded-incomplete, with the assumed semantics treated as an auditable object.**
