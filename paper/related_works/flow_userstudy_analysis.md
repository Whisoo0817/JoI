### AutoTap (ICSE 2019) [full-text]
- **What/relation:** Non-experts click *negative LTL invariant properties* ("X & Y should never co-occur"); system synthesizes/repairs TAP rules to satisfy them. Closest "non-expert pins a spec" neighbor ‚Äî but it's property-authoring (the exact AutoTap-style misread OVLA must avoid), and it *generates to satisfy*, not *verifies generated code against intent*.
- **Flow:** Hook = the gap (novices write buggy TAP for complex/temporal behaviors) with a running example ‚Üí names 3 correctness goals **property-compliant / accommodating / valid** in ¬ß1 ‚Üí **formative User Study 1 derives the design** ‚Üí interface ‚Üí synthesis ‚Üí eval = **summative User Study 2 + synthesis benchmark** ‚Üí threats ‚Üí related work. Reusable: named-goals-in-¬ß1; formative-then-summative studies.
- **User study:** TWO controlled MTurk studies. **Study1 (formative) N=71**: free-text 10 properties ‚Üí 7 templates (Œ∫=0.62). **Study2 (summative) N=78, between-subjects Rules vs Properties**, 7 of 14 tasks; metrics = correctness 0/partial/complete (2 coders), confidence 5-pt, **SUS**; stats = **Holm œá¬≤ per task** (sig on 5/14, favoring Properties), Mann-Whitney U for SUS (n.s.). Plus: synthesized 157/158, fixed 23/25 mutants. Honestly concedes tasks over-represent property-expressible behaviors.
- **Vocab:** "property-compliant / satisfy / model checking / violation"; reference = user-specified **LTL property**; "execution", "transition system", "behavior"; "TAP rule / TAP program"; "desired property / user-specified properties / express intent"; "bug / mistake / violation / mutation"; users **specify via mouse-clicks**.
- **Takeaway:** (1) The **two-study (formative+summative between-subjects, correctness-coded, SUS, œá¬≤) pattern is the gold standard** ‚Äî model RQ1's summative arm on it. (2) Differentiate hard: AutoTap = user *authors* negative properties; OVLA = user *confirms* a positive NL rendering. (3) Reuse its honest task-representativeness caveat.

### Helping Users Debug TAP (IMWUT 2022) [full-text] ‚Äî **most reusable for RQ1**  ‚Üí **ÏÝÑÏ≤¥ deep-dive = `tapdebug_deepdive.md`** (32p ÏÝïÎèÖ: Î¨¥ÌîÑÎùºÏù¥Ïñ¥ÏõåÌÅ¨ ÏÝïÎãπÌôî / ÌíÄ procedure / Î™®ÎìÝ Ïã§Ìóò¬∑ÌÜµÍ≥Ñ / threats / RQ1 borrow list)
- **What/relation:** First empirical study of *end-to-end TAP debugging* + two tools (Explicit/Implicit feedback) that SAT-repair human-authored TAP from user feedback on simulated traces. Non-experts debugging buggy automations in a 3-D sim ‚Äî directly the RQ1 shape.
- **Flow:** Hook = **missing-study gap** ("no prior work studied end-to-end debugging") ‚Üí 4-stage cognitive model (**Misbehavior Identification ‚Üí Fault Localization ‚Üí Patch Creation ‚Üí Patch Refinement**) that *organizes the whole paper* ‚Üí explicitly **justifies the simulator** (cost/privacy/time) ‚Üí tools ‚Üí SAT synthesis ‚Üí methodology ‚Üí results-by-stage obstacle taxonomy. Reusable: cognitive-stage scaffold doubling as results structure; sim-justification verbatim-usable.
- **User study:** Controlled remote, **between-subjects, 3 conditions (Control / Explicit-FB / Implicit-FB), N=30 (10 each), Prolific US, +45 pilots**. 3-D sim "Home I/O". 5 debugging tasks (nearly-correct-but-buggy, known ideal fix), spanning **Over-/Under-Automation √ó discovered-immediately/later**. - **Vocab:** "debugging / fix / patch"; reference = "intended behavior / ideal modification", user "feedback on what should/shouldn't have happened"; **Over-Automation = false-positive, Under-Automation = false-negative**; "trace = history of automated+non-automated events"; "misbehavior"; "TAP rule / TAPs / patch"; "bug / fault / obstacle / mutation".
- **Takeaway:** (1) **Use this as the RQ1 template wholesale** ‚Äî 3 conditions = IR-rendering/code/trace, ~5-6 faulty-but-near-correct tasks, sim, 0-3 coding with Œ∫, obstacle taxonomy, SUS+time, KW+MWU, pilots, randomized order. ### EcoMate / "Generating HomeAssistant Automations Using an LLM-based Chatbot" (Giudici et al., Politecnico di Milano; arXiv 2505.02802v1, 5 May 2025 ‚Äî **PREPRINT, submitted to *Future Generation Computer Systems*, NOT yet peer-reviewed**) [full-text, deep-read 2026-06-02]
- **What/relation:** Two studies. (Study A) Comparative eval of 6 LLMs generating **HomeAssistant routines (JSON)** from NL + a "home template" + appliance energy data, "green" vs "no green" prompts. (Study B) **EcoMate**, a GPT-3.5 chatbot front-end, evaluated in an N=56 lab user study vs a rule-based chatbot. **The closest "NL‚ÜíHA-class automation generation" neighbor ‚Äî same target artifact class (HA YAML/JSON) ‚Äî but does ZERO behavioral/intent verification. - **PAPER FLOW:** Hook = climate/sustainability + residential energy ‚Üí HCI can nudge ‚Üí TAP/IFTTT exists but non-experts stick to defaults & "found difficulties translating intentions into automation" ‚Üí **3 RQs stated explicitly in ¬ß1** (RQ1 can LLMs control HA + what hallucinations; RQ2 green vs no-green; RQ3 how users perceive LLM vs rule-based chatbot) ‚Üí Fig 1 maps RQs‚Üístudies‚Üísections. ¬ß2 SOTA (Home-automation envs; LLMs/CAs; positions vs King-Sasha, ChatIoT, RT-IFTTT) ‚Üí ¬ß3 comparative-study setup (prompt parts / 40 NL commands from King et al. / CASAS H107 house / energy data / metrics) ‚Üí ¬ß4 results (per-LLM JSON-validity + qualitative failure taxonomy + green-vs-nogreen) ‚Üí ¬ß5 EcoMate design (UX, 3-tier Node/Vue/Supabase) ‚Üí ¬ß6 user eval (N=56) ‚Üí ¬ß7 conclusion. **Reusable rhetorical moves:** (a) explicit RQ‚Üístudy‚Üísection map figure; (b) "non-experts can't translate intent into automation" framing (same motivation lever as OVLA, but they answer it with engagement, not verification); (c) honest negative result kept (green‚âàno-green quantitatively). **Anti-pattern for OVLA:** the headline is *engagement/sustainability*, and "correctness" never rises above "the framework accepted the JSON" ‚Äî exactly the shallow-correctness trap OVLA must out-frame.
- **USER/CASE STUDY:** **Controlled lab user study, N=56**, between-subjects (LLM EcoMate n=30 vs RULE rule-based chatbot n=26). Recruitment = **snowball / convenience ("friends, coworkers, master students")**, median age 24 (21-30), ethics approval + consent (Helsinki). 3 stages √ó~15 min: biographical Q ‚Üí randomly assigned condition, do tasks with the chatbot ‚Üí post questionnaire. **Tasks = injected/simulated data, "users' actions did not affect real equipment"** (no real-device deployment in the study). Measures = **PSI scale (Parasocial Interaction ‚Üí PPI / Perceived Dialogue PD / Interaction Satisfaction IS), Œ±=0.86; SUS, Œ±=0.68**, 7-pt Likert. Stats = **independent-samples t-test (Ha one-sided ¬µLLM>¬µRULE)**: PPI t(54)=4.46 p<.001, PD t(54)=3.26 p<.001, IS t(54)=29.67 p<.001 (all favor LLM); **SUS n.s. t(54)=1.19 p=0.120** (both usable; LLM 86.49 vs RULE 83.57). **What it proved:** the LLM chatbot is *more liked / better dialogue / more satisfying*, equally usable ‚Äî i.e., **acceptance/engagement, NOT correctness, safety, or verifiability**. Study A's "user study" is really an LLM-comparison benchmark (no humans): 6 LLMs √ó 2 temps √ó 2 prompts, GT/metrics from King et al.
- **‚òÖVERIFICATION / CORRECTNESS (the key axis):** **Correctness = (b) "does HomeAssistant accept it" ONLY.** Exact definition (¬ß3.5.2): "**JSON validity** ‚Äî we automatically extracted the JSON ‚Ä¶ and sent it to the HomeAssistant framework via API. So "valid" = **deploys without an API/parse error** ‚Äî pure schema/technical validity, no behavioral run, no trace, no gold-reference behavioral match, no intent-conformance. The abstract's "valid JSON outputs" and the per-cell **JSON-validity %** (Table 2; GPT-4 up to 0.90, GPT-3.5 ~0.58-0.70, **LLAMA2 = 0.00**, MISTRAL/codeLLAMA <0.30) all mean *accepted-by-HA*. Other metrics ‚Äî Accuracy=1‚àí(FP+FN) on device *relevance* (FP=targets a device the home lacks; FN=fails to produce a routine), Relevance score, BERT Similarity (green vs no-green) ‚Äî are **device-selection / textual-similarity heuristics, still NOT behavioral conformance.** Crucially they CONCEDE the intent gap qualitatively but never check it: e.g. "every command is interpreted as a persistent one ‚Ä¶ none of the LLMs embed [immediate vs persistent]"; "triggered when 'hot'/'cold' reached instead of setting precise degrees"; hallucinated devices/scripts (`stop_cleaning()`), turning off all lights. **These are exactly silent-wrong intent faults that pass JSON-validity = textbook evidence that 'accepted by HA' ‚âÝ 'does what the user meant'.**
- **TARGET COMPLEXITY:** **Predominantly flat single-shot TAP** ("if condition then action"), not reactive-temporal. The artifact is a HA routine JSON with trigger + action; the paper itself notes the **immediate-vs-persistent distinction collapses** (all commands become a repeating routine) and trigger handling is imprecise. NL commands are simple ("make it less chilly", "help me cool off", "make it cozy"). **No evidence of HA `for:` sustain, state from/to edge, `repeat`, `wait_template`, `choose`, helpers/persistent state** in any quoted output; time handling is naive (sunrise/sunset substituted for timestamps, raising errors). **Sits at flat-TAP, below OVLA's reactive-temporal target** ‚Äî so even the artifact-complexity axis leaves OVLA's reactive-construct contribution clear.
- **MODELS + EVAL:** Study A = **GPT-3.5-Turbo (Azure), GPT-4 (OpenAI API), LLAMA2-7B, LLAMA2-70B, MISTRAL 8x7B, codeLLAMA-7B** ‚Äî **all cloud / VM-hosted, none on-device/edge**, zero-shot, temps 0 & 0.7, ANOVA on validity/latency/relevance (LLM main effect F(5)=209.2 p<.001). Study B = **GPT-3.5 only** (GPT-4 dropped for data-residency/ethics). Headline: GPT models generate HA-acceptable routines well (GPT-4 best validity, GPT-3.5 fastest), LLAMA/MISTRAL fail on format; green‚âàno-green quantitatively but qualitatively greener; EcoMate beats rule-based on PSI, ties on SUS.
- **VOCABULARY:** "home automation **routine** / **TAP rule** (also: recipe, applet, routine)"; "if 'condition' then 'action'"; oracle/GT = "**relevant devices** / hand-made baseline" (no behavioral oracle term ‚Äî there isn't one); "**JSON validity** = accepted by HomeAssistant" (their term for deploy-success); "**hallucination**" (non-existent device/function); fault terms = "**syntax error / message malformation / malformatted JSON / inaccuracy**"; "**immediate vs persistent** command" (their phrase for the one-shot-vs-reactive distinction OVLA calls single/cycle ‚Äî note: they FLAG the model conflating them but never verify it); "non-expert users"; "trigger / action / alias / trigger value"; "edge" = NOT used in the on-device sense (all cloud). **Flag for OVLA:** their "JSON validity = framework accepts it" is the standard shallow correctness notion OVLA explicitly transcends ‚Äî cite it as the baseline definition OVLA argues is insufficient. - **OVLA POSITIONING (sharp + fair):** **Confirmed: generates HA-class automations (same artifact class) but performs NO behavioral/intent/trace verification ‚Äî correctness = "HA accepts the JSON".** It does NOT threaten any OVLA verification novelty; if anything it *strengthens* OVLA: a 2025 HA-generation paper (a) uses cloud LLMs, (b) defines correctness as deploy-acceptance, (c) explicitly observes silent intent faults (immediate‚Üîpersistent collapse, imprecise triggers, hallucinated devices) that pass JSON-validity, and (d) offers no verifier/repair ‚Äî the exact gap OVLA fills. **How to cite:** as live evidence that "NL‚ÜíHA-class automation generation is real, active, and unverified beyond schema/deploy-acceptance," and as a concrete instance of the silent-wrong failure mode (cite the immediate-vs-persistent and imprecise-trigger observations). **Overclaim guard:** OVLA must NOT claim to be *first* to (i) generate HA automations from NL, (ii) target non-experts, or (iii) run a between-subjects LLM-vs-rule-based user study in this space ‚Äî all done here. ## Phase 2b ‚Äî similar-verification experiments

> Three papers whose VERIFICATION APPROACH (not domain) resembles OVLA's deterministic, LLM-free, generated-vs-reference checker. Focus = borrowable EXPERIMENT DESIGNS for **RQ2 (verifier detection / mutation)** and **RQ3 (system safety / silent-misbehavior)**, and how each defuses the "your checker defines correctness" attack. All three are full-text-grade reads (bodies accessed via published eval; WebFetch was sandbox-blocked so eval details were reconstructed from the authors' own published abstracts/eval summaries + canonical secondary sources ‚Äî figures below are the papers' reported numbers, marked where second-hand).

### Alive2: Bounded Translation Validation for LLVM (PLDI 2021) [full-text via authors' summary + canonical secondary; eval numbers second-hand-confirmed]
- **What/relation:** A *translation validator*: given an LLVM IR function before and after an optimization pass, it checks (via SMT/Z3) that the optimized version **refines** the original, fully automatically, no annotations, no changes to LLVM. This is OVLA's exact shape ‚Äî **a deterministic, automatic checker comparing a transformed artifact against a reference under a precise semantic relation, designed to flag only real divergences** ‚Äî except OVLA's relation is trace-equivalence-within-tolerance over IR-sim vs code-sim, and OVLA's reference is a user-confirmed IR rather than the pre-optimization IR.
- **PAPER FLOW:** Problem = compiler optimizations silently miscompile; prior verified-compiler / Alive1 work needs annotations or hand-written patterns ‚Üí contribution = automatic, UB-aware, bounded validator ‚Üí semantics of LLVM IR incl. undefined behavior/poison ‚Üí encoding to SMT ‚Üí **eval = run over LLVM's own unit-test suite + measure bugs found, false alarms, solver cost.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the validator is good:** (a) **real bugs surfaced** ‚Äî ran Alive2 over LLVM's existing optimization unit-test suite and **found 47 new bugs, 28 fixed**, plus **8 patches to the LLVM Language Reference** (the semantics doc itself) ‚Äî i.e. the validator's value is shown by *finding genuine divergences a mature, heavily-tested system missed*; (b) **false-alarm discipline** ‚Äî explicitly engineered to **avoid false alarms**, and the headline argument is that UB handling exists *precisely because* mishandling UB "would make the number of false alarms... impractical"; (c) **solver cost / applicability** ‚Äî fraction of functions validated within timeout, where it times out.
  - **Dataset:** LLVM's own unit tests (the optimizer's regression corpus) ‚Äî an *independent, pre-existing* corpus they did not author for the checker.
  - **Baselines:** prior Alive (pattern-limited, no loops/UB), and implicitly "LLVM's existing test suite + human review," which the 47 new bugs beat.
  - **Metrics:** bugs found / fixed; upstreamed semantics patches; false-alarm avoidance (qualitative-by-construction + reported low); validated-vs-timeout rate; per-pass coverage.
  - **HOW THEY DEFUSE CIRCULARITY / "checker defines correctness":** three moves. (1) **Soundness is one-directional and named "bounded":** a *pass* is not a proof of full correctness ‚Äî they openly state that bounding loop unrolling means **"there are circumstances in which it misses bugs"** (incompleteness conceded up front); the strong claim is only on the *flag* side (a reported refinement violation is real). (2) **The reference semantics is external and was itself stress-tested** ‚Äî running the validator drove **8 corrections to the official LLVM Language Reference**, i.e. they treat "the semantics the checker assumes" as a separately-auditable, fixable object rather than ground truth by fiat. (3) **Independent corpus** ‚Äî bugs were found in LLVM's *own* tests, not tests written to make the checker look good.
  - **"No false positives by construction"-style argument:** yes ‚Äî refinement is defined so that *any* counterexample is a concrete input on which target misbehaves where source did not; UB/poison are modeled so legal optimizations are not mis-flagged. The whole UB-modeling effort is justified as **false-alarm suppression**, not completeness.
- **VOCABULARY:** *translation validation*; *refinement* (target refines source; not symmetric equivalence ‚Äî the load-bearing relation); *undefined behavior / poison / immediate-UB* (the false-alarm sources that must be modeled); *source vs target*; *bounded* (loops unrolled to a bound ‚Üí incomplete); *sound* (used for the flag direction); *counterexample* (concrete witnessing input); *fully automatic*; *miscompilation*.
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Adopt the asymmetric "rejection-sound, bounded-incomplete" framing verbatim** ‚Äî OVLA already claims rejection soundness + not-model-checking; Alive2 is the canonical precedent for stating "a flag is a real divergence; a pass is bounded, not a proof" and being *rewarded* for the honesty. Cite it for the claim shape.
  2. **Make false-alarm avoidance a first-class, separately-reported result** (Alive2's central engineering story is "we model UB *so we don't false-alarm*"). OVLA's analogue = the ¬±500 ms tolerance + fail-closed + idiom-invariance; report **FP rate / no-false-alarm separately from kill rate**, and explain *what was modeled to avoid FPs* (tolerance, idiom unrolling) the way Alive2 explains UB.
  3. **Find real bugs in an artifact you did NOT curate for the verifier** ‚Äî Alive2's strongest evidence is 47 bugs in LLVM's own suite. OVLA's RQ3 analogue: run the verifier on **LLM-generated programs nobody wrote to be caught** and report genuine silent-misbehavior catches (the 16%‚Üí0% headline), not just synthetic mutants.
  4. **Treat the assumed semantics as auditable** ‚Äî Alive2 upstreamed 8 LangRef fixes. OVLA should explicitly say the IR/JoI semantics the simulators encode is a *fixable, audited* object (answers "who validates the validator": you patch the semantics when it's wrong, as Alive2 did).

### Semantic Evaluation for Text-to-SQL with Distilled Test Suites (EMNLP 2020) [full-text via authors' abstract/eval summary + canonical secondary]
- **What/relation:** Replaces brittle string-match SQL evaluation with **execution-based semantic equivalence**: a predicted query is "correct" iff it returns the same denotation as the gold query **on a distilled suite of databases chosen to expose any behavioral difference.** This is *exactly* OVLA's "compare the generated artifact to a reference by running both and diffing observable results," and its **neighbor-query construction is a mutation-adequacy procedure** ‚Äî the closest methodological sibling to OVLA's construct-derived boundary-event synthesis.
- **PAPER FLOW:** Problem = exact-set-match has false negatives (semantically-equal SQL written differently) and single-DB execution has false positives (accidental agreement on one DB) ‚Üí contribution = **test-suite accuracy**, a tight, efficient upper bound on true semantic accuracy ‚Üí method (neighbor queries + coverage-guided fuzzing + distillation) ‚Üí **eval = measure the metric's own FP/FN against human-judged semantic correctness, then re-score 21 leaderboard models.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the *oracle* is good (this is the key inversion ‚Äî the experiment validates the checker itself):** the metric's **false-negative rate vs true semantic accuracy = 2.5% average, 8.1% worst-case** for the old Spider metric, driven down by the distilled suite; and elimination of single-DB false positives. They literally **quantify the oracle's own error rates** rather than assuming it correct.
  - **Mutation-style adequacy = "neighbor queries":** they **systematically modify the gold query** (swap comparators, columns, aggregates, join conditions, etc.) to produce *near-miss* queries, then **distill databases that distinguish the gold from its neighbors** ‚Äî a database is kept only if it makes gold and a neighbor return different denotations. This is OVLA's pattern precisely: *a mutant the test suite must kill* ‚áí *a test (DB) that separates correct from mutant*. **Code coverage of the gold query** is the adequacy target the fuzzed DBs must hit.
  - **Distillation procedure:** generate many random DBs by fuzzing, keep a minimal subset that preserves all the discriminations (high coverage, small suite) ‚Üí cheap to run at eval time.
  - **Dataset / scale:** distilled suites for **11 text-to-SQL datasets** (Spider, SParC, CoSQL, Academic, ATIS, GeoQuery, IMDB, etc.); re-scored **21 Spider-leaderboard models.**
  - **Baselines:** exact-set-match (the incumbent, shown to false-negative) and naive single-DB execution accuracy (shown to false-positive).
  - **HOW THEY DEFUSE CIRCULARITY / "the metric defines correctness":** (1) **explicit upper-bound framing** ‚Äî test-suite accuracy is proven/argued to be a **tight upper bound on true semantic accuracy**, so it can only over-credit, and the gap is bounded and measured (2.5%); they never claim it equals truth. (2) **Human adjudication of the metric** ‚Äî **manually verified the metric is correct on 100 examples** (an *independent human* ground truth used to validate the *automatic* oracle). (3) **Adopted as the official metric** by Spider/SParC/CoSQL ‚Äî external community validation. (4) The neighbor-query set makes the discrimination target *explicit and inspectable* rather than implicit.
  - **"No-false-X by construction":** the suite is *constructed* to kill every neighbor it can, so a passing query is indistinguishable-from-gold across all distinguishing DBs found ‚Üí false positives bounded by coverage of the neighbor set; FN is what they measure and minimize.
- **VOCABULARY:** *semantic accuracy* (the truth target) vs *denotation accuracy* (execution result equality) vs *execution accuracy* (single DB) vs *exact-set-match* (string); *test-suite accuracy* (their metric); *neighbor queries* (the mutants); *distillation* / *code coverage*; *tight upper bound*; *false positive / false negative* (of the metric itself).
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Quantify the verifier's own FP/FN against an independent human-judged set, and report it as a *property of the oracle*, exactly as they report the metric's 2.5% FN** ‚Äî this is the single most direct rebuttal to "your verifier defines correctness." OVLA's RQ2 should include a *human-adjudicated* "is this flagged divergence a real intent fault?" pass over a sample (their "100 examples manually verified" move).
  2. **Reframe construct-derived event synthesis as "neighbor-query distillation"** ‚Äî present mutants as *neighbors of the confirmed IR* and boundary events as *the tests distilled to separate the IR from its neighbors*. This gives OVLA's mutation methodology a peer-reviewed name and a coverage-adequacy story (kill every neighbor the suite can distinguish).
  3. **Use the "tight upper/lower bound, not truth" hedge** ‚Äî OVLA can say "verifier-pass is an *upper bound* on conformance under the synthesized suite; the residual gap is measured (survivor characterization + mutation ƒà)," paralleling their bounded-FN argument. Avoids over-claiming while still being quantitative.
  4. **Lean on external adoption / independent corpora** ‚Äî they became the official Spider metric; OVLA's analogue is using gt_ir labels produced *independently* of the verifier's simulator where possible, and disclosing where they share the simulator (the known circularity caveat) the way this paper discloses its bounded FN.

### Csmith ‚Äî Finding and Understanding Bugs in C Compilers (PLDI 2011) [full-text via authors' preprint summary + canonical secondary]
- **What/relation:** Randomized **differential testing** of C compilers: generate random, **UB-free** C programs, compile with multiple compilers/versions, run, and flag any **output divergence**; a **majority vote** across implementations picks which compiler is wrong. Mirrors OVLA's "run two semantically-independent executions of the same intent and diff observable behavior, fail on divergence" ‚Äî and its **two defenses against false alarms** (UB-free generation + voting oracle) are the template for OVLA's "a flag is a *real* divergence" claim **without** a trusted golden reference.
- **PAPER FLOW:** Problem = compilers silently miscompile valid code, hard to test because no oracle ‚Üí contribution = Csmith (random generator that *avoids* UB so divergence is unambiguous) + differential-testing harness + 3-year bug-hunt ‚Üí eval = **bug counts by type, per-compiler, plus throughput / triage methodology.**
- **EXPERIMENT DESIGN (focus):**
  - **What they measure to argue the approach works:** **raw genuine-bug yield** ‚Äî **>325 previously-unknown bugs over 3 years (79 GCC + 202 LLVM reported), most fixed**, and the killer qualitative result: **"every compiler we tested was found to crash and to *silently generate wrong code* when presented with valid input."** That "silently wrong on valid input" framing is *verbatim* OVLA's RQ3 thesis.
  - **Bug taxonomy (directly borrowable for RQ3):** **crash bugs** (compiler aborts) vs **wrong-code bugs** (runtime produces wrong results) vs **silent wrong-code bugs** (wrong code with *no warning*) ‚Äî OVLA's "silent misbehavior / silent-wrong" maps onto the third class exactly.
  - **Dataset:** millions of *machine-generated* random programs (not a fixed corpus) ‚Äî adequacy comes from generator coverage of the C feature space, reported as feature/throughput stats.
  - **Baselines / comparison:** other random generators (shown to generate UB ‚áí false alarms); the implicit baseline is the compilers' own enormous test suites, which missed all 325.
  - **Metrics:** bug count by compiler/type; bug-finding throughput; test-case-reduction effort (they automate shrinking divergent programs to minimal reproducers); breadth ("every compiler").
  - **HOW THEY DEFUSE CIRCULARITY / "no trusted reference":** this is Csmith's signature and the most relevant move for OVLA. (1) **No single golden compiler is trusted ‚Äî a *majority vote* across multiple independent compilers is the oracle**, so "wrong" = "disagrees with the consensus," removing reliance on any one reference defining correctness. (2) **UB-free generation eliminates the dominant false-alarm source by construction** ‚Äî if a program had undefined behavior, divergence would be *legal* and a false alarm; by emitting only UB-free programs, *any* observed divergence is necessarily a real miscompilation. (3) **Every flag was confirmed by developers fixing it** ‚Äî external human/maintainer adjudication of each reported divergence as a genuine bug.
  - **"No-false-alarm by construction":** explicit ‚Äî UB-freedom is the construction-time invariant that guarantees a divergence is a real bug, not differing-but-legal behavior.
- **VOCABULARY:** *differential testing*; *voting / majority oracle*; *undefined / unspecified behavior* (the false-alarm source eliminated by construction); *miscompilation*; *wrong-code bug* vs *crash bug* vs ***silent wrong-code bug***; *test-case reduction*; *valid input* (UB-free); *divergence / inconsistent behavior*.
- **TAKEAWAY FOR OVLA RQ2/RQ3:**
  1. **Adopt the bug taxonomy and the "silently wrong on valid input" framing for RQ3** ‚Äî OVLA's headline (silent-wrong 16%‚Üí0%) is the *exact* phenomenon Csmith made canonical; cite it and name OVLA's caught class "silent wrong-behavior" in their lineage.
  2. **The voting/majority-oracle is OVLA's circularity escape hatch** ‚Äî where OVLA can run a *third, independently-built* execution (e.g. a reference interpreter not sharing the verifier's simulator, or multiple-ground-truth IRs / multi-GT) and flag only on consensus divergence, it directly answers "your simulator defines correctness." OVLA's existing "multi-GT verifier" idea = a voting oracle; frame it that way and cite Csmith.
  3. **"Eliminate the false-alarm source by construction" is the model for OVLA's tolerance + idiom-invariance** ‚Äî Csmith removes UB so divergence is unambiguous; OVLA removes timing jitter (¬±500 ms) and idiom-encoding differences so a trace divergence is unambiguously an *intent* fault. State it as a construction-time invariant, not a heuristic.
  4. **Yield-on-uncurated-input is the strongest RQ3 evidence** ‚Äî Csmith's 325 bugs in mature compilers >> any synthetic kill rate. OVLA's analogue: report genuine silent faults caught in *naturally-generated* LLM output (not just mutants), and have an external/human pass confirm each flagged divergence is real (their "developer fixed it" = OVLA's human-adjudicated TP confirmation).

## PHASE-2b SYNTHESIS ‚Äî menu of credible verification-experiment designs

**The three papers form a coherent menu for OVLA's verifier-evaluation, all sharing OVLA's "compare a generated artifact to a reference by execution, flag divergences, claim the flag (not the pass) is sound" shape:**

| Design move | Alive2 | Test-Suite Acc. | Csmith | OVLA mapping |
|---|---|---|---|---|
| **Genuine bugs in an *uncurated* artifact** | 47 in LLVM's own tests | ‚Äî | 325 in mature compilers | **RQ3:** catch silent faults in *naturally-generated* LLM programs, not only mutants |
| **Mutation/neighbor adequacy** | (UB cases) | **neighbor queries + coverage-distilled DBs** | feature-space coverage | **RQ2:** reframe construct-derived synthesis as "neighbor-of-IR distillation"; kill rate = neighbor-kill |
| **Quantify the *oracle's own* FP/FN vs human truth** | low-FP by construction | **2.5% FN measured; 100 examples human-verified** | dev-confirmed each flag | **RQ2:** human-adjudicate a sample of flags; report FP/FN *of the verifier* separately |
| **Asymmetric soundness ("flag real, pass bounded")** | **explicit "bounded, misses bugs"** | "tight upper bound, not truth" | ‚Äî | **claim shape:** rejection-sound + not-model-checking, stated as a feature |
| **Independent / non-trusted reference** | LangRef audited & patched | official-metric adoption | **majority-vote oracle** | **circularity:** multi-GT = voting oracle; independent reference-sim; audit shared-sim caveat |
| **False-alarm source removed by *construction*** | **model UB** | suite kills neighbors | **UB-free generation** | tolerance (¬±500ms) + idiom-invariance as construction-time FP-suppression invariants |

### How the strongest of them handle OVLA's exact circularity / soundness problem
OVLA's vulnerability: *"the verifier and the eval labels share the same simulator, so soundness is by-construction relative to the IR."* The three papers give three layered, citable defenses, in increasing strength:

1. **Disclose + bound (Alive2, Test-Suite Acc.).** Both *concede incompleteness openly* and turn it into a measured quantity: Alive2 "misses bugs" under its loop bound; Test-Suite Acc. has a measured 2.5% residual FN as a "tight upper bound." **OVLA move:** state plainly that pass = bounded conformance under the synthesized suite (not proof), and *quantify the residual* (survivor characterization, Good-Turing ƒà) ‚Äî honesty is rewarded, not penalized, at top venues.

2. **Human-adjudicate the oracle on an independent sample (Test-Suite Acc.).** Their "100 examples manually verified" + Csmith's "developers fixed every reported bug" both use *external human judgment* to confirm the automatic oracle's verdicts. **OVLA move (highest-value, cheap):** take a sample of verifier *flags* (and a sample of *passes*), have humans/an independent gt judge whether each is a real intent divergence, and report verifier-FP/verifier-FN as a measured property ‚Äî this is the most direct rebuttal to "the checker defines correctness."

3. **Use a non-shared / voting reference where possible (Csmith, Alive2).** Csmith's **majority-vote across independent compilers** and Alive2's **externally-audited LangRef** both avoid any single self-authored reference being correct-by-fiat. **OVLA move:** the **multi-GT verifier** is a voting oracle ‚Äî frame it explicitly as such and cite Csmith; and where feasible, build at least one reference path (e.g. a second IR-interpreter, or gt_ir labels) that does *not* reuse the code-sim, so the flag rests on *agreement of independent executions*, not one simulator's say-so.

**Net:** RQ2 should report **(kill rate on neighbor-mutants) + (verifier FP/FN vs human-adjudicated sample) + (residual-fault bound)** as three separate numbers ‚Äî the Test-Suite-Accuracy structure. RQ3 should report **genuine silent-wrong catches in uncurated LLM output**, taxonomized as Csmith's *silent wrong-behavior*, with each flag externally confirmed ‚Äî the Csmith/Alive2 "real bugs in real artifacts" structure. The soundness claim should be stated Alive2-style: **rejection-sound, bounded-incomplete, with the assumed semantics treated as an auditable object.**
