## Abstract (PROSE — submission, 2026-05-29; codex-iterated, claims narrowed to measured evidence)
People increasingly create home-automation rules by describing them in plain language, and a language model turns each description into rule code that then runs unattended on a hub. The problem is that this code cannot easily be checked against what the user actually meant before it goes live, and *reactive* automation makes the check especially hard: behavior plays out over time — should the rule fire once or every time the condition recurs? only after it has held for a while? — so the code is temporally opaque, there is no single output to compare against, and non-developer users cannot write a formal specification themselves. Prior NL-to-automation work leaves this check outside the deployment loop — to post-hoc human inspection of the generated rule, to an offline expert who hand-writes a correctness property, or to a cloud LLM re-check — each of which needs a person or the cloud in the loop at authoring time. That assumption fails on the **edge**, where rules are generated and deployed locally for privacy and latency with no human reviewer or cloud judge available, so the check must instead be cheap, deterministic, and on-device. We present **VERA**, an on-device pipeline for making LLM-generated edge automations checkable before deployment. VERA first translates a user's natural-language command into a typed **Timeline IR**: a compact representation that names the rule's temporal behavior explicitly — when a condition starts, how long it must hold, whether the action repeats, and when it stops. VERA renders this IR back into plain language for the user to confirm. After confirmation, VERA treats the IR as the specification: it generates reference event traces and boundary cases from the confirmed behavior, then uses an **LLM-free verifier** to check whether the generated program (in an existing reactive-automation DSL, JoI) matches those traces; if it fails, VERA shows the counterexample to the generator and retries. We evaluated VERA on **382 reactive automations across 24 categories**. In the deployment setting — where the user has already confirmed the IR and VERA only has to translate that confirmed behavior into code — the verifier **flagged every faulty program (no false negatives on this set)**; counterexample-guided retry then **repaired 32% of them with no regressions**, and the rest were rejected rather than deployed. To stress-test the verifier itself, we injected bugs spanning the IR's constructs — wrong thresholds, added or dropped actions, inverted conditions, off-by-one timing, missing one-shot stops — and it **rejected 1461 of 1475 faulty programs (99.0%)**, with the tests exercising **97.4%** of the targeted boundary cases. Against $X$, VERA improves end-to-end correctness by $\Delta$ while keeping the entire check on-device.¹ Running on a **4-bit ≤9B local model, with no fine-tuning and no LLM in the verification loop**, VERA shows that LLM-generated reactive automation can be made checkable against a user-confirmed specification before it is deployed — entirely on the edge.

> ¹ *$X$ (baseline) and $\Delta$ (improvement) are placeholders, filled after today's baseline experiment.*

## Abstract (flow — working notes)
• [PROBLEM] End-user reactive automation is moving to natural language, and on-device sLLMs (≤9B) are the only privacy/latency-viable generator — but the generated rule-code *cannot be verified against the user's intent before deployment*.
• [WHY EXISTING VERIFICATION DOESN'T TRANSFER] NL→code checking works when some *oracle* is available: a gold result or test/example suite for executable comparison, or a human-authored property/monitor that defines correctness. Reactive automation has none of these at deployment time: outputs are *future device side-effects over event sequences* (no on-demand comparison); its idiomatic rule code is *temporally opaque* and not reliably inspectable (edge-vs-level/re-arm not on the surface; an LLM round-trip to NL has no faithfulness guarantee and yields no executable oracle); and *non-developer end users cannot author temporal formal specs*.
• [IDEA — hero contribution: IR-as-spec] We introduce **Timeline IR**, a *normalized behavioral representation* for a scoped fragment of reactive-temporal automation: a fixed set of eight typed step operators (plus field-level extensions) expresses that fragment's reactive constructs (P1–P14) as typed fields rather than target-code idioms, so the confirmed behavior is stable across multiple valid JoI encodings. The IR renders deterministically to NL for **user approval** (confirm-by-approval — no formal expertise; the right bar for non-developer smart-home users); after confirmation, **the IR is the specification — instantiating the oracle the domain otherwise lacks**. From it we derive **reference traces (IR-SIM)** and **construct-derived boundary obligations (IR-FSM)**, letting an **LLM-free verifier check the candidate JoI against the confirmed IR** by *bounded trace conformance* (Rung 1) — turning trace comparison into a structured, diagnosable check rather than an ad hoc diff.
• System: a **decomposed NL→IR→JoI pipeline** on a ≤9B on-device model — intent analysis, service & device mapping, IR assembly, lowering, with **deterministic glue at every join (no autonomous agent; LLM-free verdict)** — plus a two-projection verifier (IR-SIM trace oracle, IR-FSM obligation oracle) performing **bounded behavioral conformance checking under a stated observation model**, driving deterministic accept/reject + retry. (codex: positive contract framing, NOT "never sound acceptance".)
• Results headline [DATA]: E2E accuracy vs (a) cloud-direct, (b) cloud self-repair, (c) **fine-tuned-SLM baseline**; verifier rejection power (mutation-adequacy [DATA]%), construct-boundary coverage [DATA]%; real-hub deployment [DATA]; user study — users catch [DATA]% more intent errors on IR than on raw code.
• Contribution framing: first to make LLM-generated reactive automation *behaviorally verifiable against a user-confirmed, non-developer-approvable spec* on the edge — without any human-authored formal specification.

## §3 The Spec Gap (problem formalization) (flow)
• §3.1 Three audiences ask "is it right?" (user / system / auditor) — none has an oracle.
• §3.2 Why NL can't be the spec (ambiguity); why generated code can't be the spec (idiom non-canonicality — no RA-DSL first-class-complete so idioms always remain; and no *independent* reference behavior to compare against).
• §3.3 **[CLAIM, the memorable framing] Why reactive automation can't borrow text-to-SQL's verifiability.** NL→code verification works wherever a domain supplies an oracle; reactive automation supplies none of the prerequisites:
  | Prerequisite for automated verification | text-to-SQL / code-gen | reactive automation |
  |---|---|---|
  | independent executable *reference* for intended behavior | ✅ gold query result / unit tests | ❌ rule runs on a hub, but no independent reference behavior |
  | inspectable output the user confirms | ✅ result rows shown on demand | ❌ output = *future, event-triggered* device side-effects; not on-demand |
  | user-suppliable I/O examples | ✅ PBE / tests | ❌ behavior is temporal over event sequences; user not expected to give exhaustive traces |
  | deploy-time reference | ⚠️ gold needed (offline) | ❌ novel command, no gold |
  • **Execution ≠ verification**: an emulator executes the candidate (what it does), but is not an oracle (does it match intent). The hard, missing piece is the *reference behavior* + *temporal inputs*.
• §3.3b **[CLAIM, the deepest reason no oracle exists] The reference cannot be inferred from NL alone — the missing information is in the input channel, not the model.** (1) For reactive-temporal commands, part of the intended behavior is *not entailed by the utterance*: "if the door opens, turn on the light" underspecifies one-shot vs recurring; "turn it off after a while" underspecifies the delay. (2) The lexical conventions a generator applies ("if"→one-shot, "whenever/every time"→recurring) are **priors, not truth** — a reasonable default reading, never a guarantee about *this* user's intent. (3) **A larger or better model does not close this**: scale sharpens the *prior over likely readings*, but cannot *observe* a temporal choice the user never put in the utterance — so this is a principled limit of the input, not a capability gap [pre-empts the "just use a bigger LLM" objection]. (4) Therefore confirmation is not a usability nicety but the operation that **instantiates the oracle**: it turns one underspecified command into a single executable reference specification, which is exactly the reference the table above lacks. (5) Only *after* confirmation do IR-SIM (reference traces) + scenario synthesis (temporal inputs) make trace-equivalence decidable. [the confirmation point must be the IR, not the generated code — §4, §6]
• §3.4 **Why existing verification doesn't close it (PRECISE — not "reactive automation can't be verified"; it can be *tested*)**: the active IoT-verification literature (TAPInspector'22, TAPFixer'24, HAWatcher'21, Soteria/IoTSan) verifies **safety / rule-interaction / anomaly** properties on *human-authored* rules; structural diagnostics (AWS IoT Events "Run analysis", HA schema, TaskSense DAG) check well-formedness. **Trace-based systems use traces the *other* direction**: Trace2TAP / "Helping Users Debug TAP" / HAWatcher / HA-Trace consume *actual* past traces to debug / repair / mine / explain **what happened** — post-hoc, over deployed rules. **We derive an *expected* trace from a confirmed spec and check generated code against it — *what should happen*, pre-deployment.** (One-liner: *prior work uses traces to explain or repair what happened; we use traces to verify what should happen.*) **Supporting precedent (not a threat)**: expert test harnesses prove reactive automation *can* be tested when someone supplies scenarios + assertions — the strongest, openHAB JRuby RSpec + `timecop`, even asserts `for:`-sustain timing offline — **but the assertions are expert-authored; NL-generated automation has none at deploy time.** **To our knowledge no system provides the conjunction we call T6**: an NL command → a *user-confirmed executable behavioral spec* → *auto-synthesized temporal scenarios* → *trace-equivalence* of the *LLM-generated idiomatic code* against the spec's expected traces. Other obvious fixes also fail: code→NL round-trip (the code is *temporally opaque* — edge-vs-level/re-arm not on its surface; an LLM round-trip is partial, has *no faithfulness guarantee* (another ungrounded LLM → the thing-to-confirm needs confirming), and yields *no executable oracle* for trace-equivalence, only an NL string); model checking (no φ, no formal target semantics, state explosion); monitor synthesis (hand-authored LTL). [full treatment §10]
• §3.4b **[CLAIM] Why the confirmable artifact is a spec, not a raw trace** (motivates confirming the IR, §4/§5): the oracle gap is *not* closed by simply showing the user a trace to approve. A raw trace is long, time-indexed, low-level, shows only **one** scenario, and gives no coverage guarantee — visualization (HA-Trace, TAP-debug history views) helps a user understand *what happened* but cannot let a non-expert certify the *full temporal behavior* across edge cases (sustain boundary, reset, one-shot-vs-repeat, all/any). Hence we split the two roles: the user confirms a *compact behavioral spec* (small, readable, complete for the scoped IR fragment), and the system — not the user — handles the *many* traces needed for verification by deriving them from that confirmed spec.
• §3.5 [CLAIM] The missing piece = an intermediate artifact that is NL-confirmable AND machine-checkable — from it we *derive the deploy-time reference oracle* the domain lacks, and from it the system synthesizes the verification traces the user never has to read. The rest of the paper builds it.

## §4 Requirements for a Confirmable Behavioral Spec (flow)
• [CLAIM] Any artifact that closes the gap must have THREE properties: **Confirmable / Executable / Obligation-derivable**.
• [TAB] property → meaning → forced-by-failure → realized-as (C1 NL-projectable, C4 simulator-interpretable, C3 FSM-derivable).
• Idiom-invariance (C2) = the cross-cutting *principle* that unlocks all three; sLLM-lowerability (C5) = deployment constraint, not a spec property.
• [CLAIM] **Confirmable is required for oracle formation, not merely usability** (§3.3b): because some temporal choices (one-shot vs recurring, the exact delay) are not entailed by the NL surface, the IR must *expose* those choices for approval *before* code generation — Confirmable collapses input ambiguity, Executable then produces reference traces, Obligation-derivable produces coverage/checks. Confirming at the IR (readable, temporal behavior visible via IR-SIM) rather than the generated idiomatic code (unreadable to the user, temporal behavior invisible) is the only sensible point — which is *why* C1 (NL-projectable) and C2 (idiom-invariance) are requirements.
• [CLAIM] Executable + Obligation-derivable are two *projections* of one IR (not two hacks) — pre-empts "why simulator AND FSM?".

## §5 Timeline IR (the artifact) (flow)
• Design philosophy: replace idiom-encoded reactive patterns with first-class typed fields (wait.edge, wait.for, cycle.count, cycle.until). **[CLAIM — scoped generality]** a *fixed set of eight typed step operators (plus field-level extensions)* covers the reactive-temporal constructs in our surveyed and evaluated RA-DSL fragment (P1–P14: edge/level/sustain/debounce/hysteresis/periodic/cron/alternation/bounded-repeat/delay-seq/state-var/timer/all-any) — whereas common consumer RA-DSLs often encode patterns like hysteresis/alternation through variables, timers, and control-flow idioms rather than uniform first-class constructs. The IR thus *normalizes* these constructs into typed fields (a normalized behavioral form, canonical within our IR grammar — not a global normal-form claim), making the confirmed behavior stable across multiple valid JoI encodings and able to serve as the single comparison reference the code cannot.
• [TAB] 8 ops + field extensions; trigger-disambiguation rule ("when"→wait edge:none; "whenever/every time"→cycle{wait rising}) — **frame as a *default interpretation applied before confirmation* (a documented prior), NOT a hard claim that the keyword fixes intent** (§3.3b); confirmation is where the user overrides it. (For the benchmark this same convention is the locked evaluation label — §9.1.)
• IR→NL projection (the confirmation channel): 1:1 deterministic rendering; one IR ⇒ one English. ## §6 IR as Spec: Two Projections (flow)
*(codex #4: §6 is CONCEPTUAL — the semantic design of why the IR can serve as spec via two projections. NO algorithms/numbers here; those live in §8. Keep §6↔§8 boundary crisp to avoid overlap.)*
• §6.1 spec role: a deterministic artifact that decides whether candidate JoI *passes the bounded conformance check* (not "is correct" — Rung-1, not completeness); "spec" = answers both "what will happen" (operational, via IR-SIM) and "which transition obligations / scenario boundaries must be exercised" (structural, via IR-FSM).
• §6.2 **IR-SIM (dynamic projection)** = executable operational semantics → reference timestamped device-service trace; **observation model defined here** = sequence of (t_ms, service, method, args); why simulator not formal semantics (JoI has none). (Definition only; the comparison algorithm + tolerance = §8.)
• §6.3 **IR-FSM (static projection)** = transition obligations (edge/sustain/sequencing/re-arm/register-dep/reachability/first-iter) derived from the control-flow tree. (What obligations EXIST; how they're covered/checked = §8.)
• §6.4 why NOT direct FSM-FSM on JoI (informal semantics, continuous state, idiom multiplicity, bisimulation gap) — SHORT, point to Related Work.
• §6.4b **[CLAIM — not model checking; the formal-verification wedge]** the IR-FSM is a *test-generation / coverage projection of the spec, not a model-checking target*: formal verification presupposes (i) a formal model AND (ii) an expert-authored property, and proves the model satisfies the property over *all* behaviors — our setting has neither (no oracle; a non-developer user), so our contribution sits *upstream* of verification (we *acquire* the reference behavior from user-confirmed intent) and our check is *concrete differential trace testing* (IR-SIM vs JoI-SIM on a finite synthesized suite), not symbolic exhaustive proof. The nearest neighbor is therefore **model-based / conformance testing**, not timed-automata model checking — and we differ from that in turn by an *intent-derived, user-confirmed* model, construct-derived obligation enumeration, and a closed self-correction loop (§8). **Honest concession**: formal verification yields all-input + liveness guarantees we do not; we trade exhaustive proof for spec-acquisition-from-intent + idiom-invariant bounded conformance — a *different problem*, not a weaker proof. [full contrast §10.3/§10.5]
• §6.5 therefore the verification *principle*: trace (IR-SIM) + obligation attribution (IR-FSM). **[CLAIM — depth, under the IR-as-spec hero]** IR-FSM *complements* the trace oracle by deriving transition obligations and coverage targets, turning trace comparison into *bounded behavioral conformance* (Rung 1) rather than an ad hoc diff. IR-SIM says "did it match on the inputs we tried"; IR-FSM says "and those inputs covered the behavior that distinguishes intent." (This is depth *within* the hero contribution — the spec is the hero; IR-FSM is what keeps the check structured, diagnosable, and bounded, not a second hero.) The *implemented* mechanism, claim ladder, coverage generator, mutation evidence, and retry are §8.

## §8 The Verifier (the moat) (flow)
• §8.1 two-layer: L1 static (over IR-FSM) → L2 trace-equivalence (IR-SIM vs JoI-SIM), short-circuit; tolerance max(500ms,10%). **Honest scoping (codex)**: L1's static-analysis *technique* is not novel — AWS IoT Events' "Run analysis" does comparable structural diagnostics (unreachable-state / missing-trigger) — though even L1 differs by running over an *intent-derived, user-confirmed* IR rather than a hand-authored FSM. The hero is **L2 = the derived reference-trace oracle (IR-SIM) + trace-conformance against a user-confirmed spec**, plus the confirmation loop. (Crucially, even L1 is *structural diagnostics, not model checking*: it seeds/filters concrete test scenarios over an intent-derived IR; the behavioral verdict is L2's concrete trace-conformance, never a symbolic property proof — §6.4b.)
• §8.2 diagnose + retry: violation → IR-feature-grained hint → sLLM retry (deterministic floor + optional LLM-aided diagnose).
• §8.3 **what we prove — the claim ladder** [CLAIM, careful — codex: frame as a CONTRACT, not an apology]:
  - **Guaranteed**: T1 Timeline-IR determinism (the *spec* is unambiguous; IR-FSM is deterministically derived from it — the IR is the spec, the FSM a projection); T2 rejection soundness — a flag ⇒ a real divergence.
  - **DEFINE "real divergence" precisely (codex warning, else overclaim)**: a divergence between IR-SIM and JoI-SIM traces, *relative to* (i) the two simulators' operational semantics, (ii) the (t,service,method,args) observation model, (iii) the max(500ms,10%) timing tolerance, and (iv) the finite fragment F (guard-discretized alphabet, bounded horizon/registers, deterministic). T2 says: within (i)–(iv), reject is never spurious. We do NOT claim "pass ⇒ correct."
  - **Covered empirically**: Rung-1 bounded transition-boundary coverage over F.
  - **Exposed**: every flag yields a concrete counterexample that feeds repair.
  - **Measured**: mutation catch rate + survivor analysis + deployment drift.
  - Abstract/intro wording = "bounded behavioral conformance checking under a stated observation model"; the "not sound acceptance" caveat lives HERE and in §11 only.
• §8.4 coverage suite (scenario synthesis): **scenario synthesis operates on the IR rather than on surface syntax**, selecting branch and operator boundary values using the IR's explicit data dependencies — happy + else-branch + flap-reset + re-arm + expression-boundary scenarios. Concretely: when a branch uses a variable produced by a sensor read, the generator traces that read back to its source sensor and injects the boundary value into that sensor's simulator state (seeding the variable name alone is inert); it targets operator-specific cases for abs/min/max and clamps; it chooses values from each sensor's declared value domain (type, range, enum); it skips self-referential read-modify-write updates that would create false trace mismatches; and it adds a second edge to test re-arming of sustained conditions. Two-sided coverage (spec-side + impl-side JoI-branch). [DATA] spec **342/351 = 97.4%** / impl Y% — this deterministic suite covers 97.4% of the targeted boundary cases without hand-authored scenarios; obligations exclude empty-else branches (no behavioral obligation on the false path); the 9 uncovered (≈3%) are guards on a cyclic counter (`n%2`) reached only by execution count, not by any injected input (§11) — NOT detection failures.
• §8.5 **mutation = rejection-power evidence** (= Contribution-2: construct-derived adequacy, target-DSL-neutral in design): **10 operators** — control-flow/temporal/comparator/assign + **expression-boundary `cmp_direction` (min↔max clamp inversion) and `arith_op` (+↔−, ×↔÷)**, equivalent-mutant filtered. [DATA] **catch 1287/1296 = 99.31%** (cmp_direction 65/65, arith_op 26/26 = 100%); 9 survivors all explained (sub-tolerance comparator + characterized coverage gap), 0 new survivors / 0 FP; tolerance-boundary sweep proves the test CAN miss (non-circular). [why expression operators matter: JoI has no abs/min/max → lowering UNROLLS to if-else, so a wrong unrolling (inverted clamp) is caught only if a scenario drives the input to the operator's internal boundary — §11 register-over-time = partial.]
• §8.6 threats to validity: L2 device-identity-blindness (tag/quantifier = precision stage, out of scope); circularity scope (JoI-SIM shared one side, independence via IR-SIM + sweep); simulator-vs-hub drift (→ deployment RQ8).
• [DIFFERENTIATE] [TAB] verification depth: GPIoT (Pass@k + SonarQube, no intent oracle) / TaskSense (dependency solvability) / monitor synthesis (hand LTL) / **ours (behavioral conformance vs user-confirmed spec, mutation-quantified)**.
• §8.x **[PLANNED] IR-SIM feasibility gate (reuses the simulator; mechanism here, policy in §7.y)**: during the existing IR-SIM run, assert hard contradictions and **reject ONLY** tick-local / schema-grounded impossibilities (same actuator assigned mutually-exclusive values in one tick; `cycle.period` < accumulated step latency; value outside device schema/enum; `if A && !A`; `break` outside cycle; non-positive wait). **Never reject on "branch never taken / wait never resolved across scenarios"** — that is a *coverage-warning* (we may not have synthesized the trigger), never an infeasibility verdict (avoids false-infeasible). Runs at IR-confirm time, system-side. Narrow static/symbolic checks only if cleanly specifiable; no full-feasibility claim without formal device models.

## Appendix A — Timeline IR Schema (Reference) *(migrated from paper_summary; canonical here)*

### A.1 Top-level
```json
{ "timeline": [ <step>, <step>, ... ] }
```

### A.2 Step grammar (8 ops + field-level extensions)
| Op | Meaning | Key fields |
|---|---|---|
| `start_at` | scenario anchor | `anchor:"now"`, or `anchor:"cron", cron:"<5-field>"` |
| `wait` | block until cond | `cond`, `edge:none\|rising\|falling`, optional `for:"<N> <UNIT>"` |
| `delay` | pause | `duration:"<N> <HOUR\|MIN\|SEC\|MSEC>"` |
| `read` | snapshot a value to a local var | `var`, `src:"<Device.attr>"` |
| `call` | device method call | `target:"<Device.method>"`, `args:{...}` |
| `if` | one-shot branch | `cond`, `then:[...]`, `else:[...]` |
| `cycle` | repeat body | `period:"<N> <UNIT>"`, optional `until:"<expr>"`, optional `count:"<var>"`, `body:[...]` |
| `break` | exit nearest cycle | — |

Field-level extensions (orthogonal to the 8-op count): `wait.for` (sustained condition), `cycle.count` (tick-indexed integer; lowering = `n:=0` + body + `n=n+1`).

### A.3 Expression grammar
- **Literals**: numbers, strings, booleans.
- **Device attribute reference**: `Device_id.attr` (e.g., `TempSensor_1.temperature`).
- **Local variable reference**: `$varname` (from prior `read`) or count variable bare name (e.g., `n`).
- **Clock**: `clock.time` (4-digit `hhmm` int), `clock.date` (`YYYYMMdd`), `clock.dayOfWeek` (`"MON".."SUN"`).
- **Operators**: `+ - * / %`, `( )`, `== != < > <= >=`, `&&`, `||`, `!`.
- **Convention β**: `args` string is an expression iff it contains `.`, `$`, or any operator; otherwise it is a literal.

### A.4 Trigger mapping (default pre-confirmation interpretation; the locked *evaluation* convention)
> These are the generator's **default readings applied before confirmation** — a documented prior, not a claim that the keyword uniquely fixes intent (§3.3b). At deployment the user overrides any row via IR confirmation. For the benchmark, this same table is the *locked evaluation label* used to make accuracy reproducible (§9.1).

| English | IR pattern |
|---|---|
| `if X, do Y` | `if` one-shot (no wait, no cycle) |
| `when X, do Y` | `wait(edge:"none") + Y` — one-shot level wait |
| `whenever X, do Y` / `every time X, do Y` | `cycle { wait(edge:"rising"); Y }` |
| `X for at least N` / `X for N` | `wait(edge:"none", for:"N UNIT")` by default |
| `every N UNIT, do Y` | top-level `cycle(period:"N UNIT") { Y }` (not cron) |
| `every N UNIT, alternate A and B` | `cycle(period:"N UNIT", count:"n") { if(n%2==0){A} else {B} }` |
| `every N UNIT, do X; stop after K times` | `cycle(period:"N UNIT", until:"n >= K", count:"n") { X }` |

### A.5 Validator rules
- Top-level: `timeline` (non-empty list). First step must be `start_at`.
- `cycle.period` required (authoritative polling cadence); `cycle.body` must contain ≥1 `delay` or edge-triggered `wait` (cadence guarantee); `cycle.count` (if present) a non-empty string identifier; nested cycles forbidden.
- `wait.edge` ∈ `{none, rising, falling}`; `wait.for` uses the `delay` duration grammar.
- Every `Device.attr` / `Device.method` must exist in the provided service catalog.

### A.6 Reject conditions
- Cycle requested but no period/interval; undefined device/attribute referenced; nested loop requested.
