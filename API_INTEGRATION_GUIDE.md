# 📘 JOI Agent API Integration Guide

## 🤖 1. 에이전트 기능

- **시나리오 생성**: generate_joi_code API.
- **피드백**: 수정 요청을 하면 이전 맥락을 기억하여 코드를 수정.
- **시나리오 관리**: 등록된 시나리오 목록을 조회하거나, 특정 시나리오를 ID 기반으로 삭제 가능. 서버의 joi_new/data/joi.db에 저장.
- **기기 상태 및 정보 조회**: 연결된 기기 파악.
- **날씨 API 사용**: 특정 지역(서울/부산/대구 등)의 날씨를 API를 통해 가져와 출력. (네트워크 연결 필요)
- **다중 대화 맥락 유지**: `session_id`를 통해 사용자별로 독립적인 대화 기록과 상태를 유지.

---

## 🚀 2. API 연동 방법

모든 기능은 `agent_chat` 호출을 통해 수행됨.

### **요청 파라미터 (Request Body)**

**`sentence`**: 사용자가 입력한 채팅 메시지 내용
**`session_id`**: 사용자 또는 브라우저 탭을 구분하는 고유 ID
**`connected_devices`**: **최초 요청 시 반드시 포함**해야 하며, 이후에는 DB에 저장되므로 생략 가능

#### **응답 데이터 (Response Body)**

**`response`**: 에이전트가 사용자에게 전하는 채팅 답변 메시지
**`last_result`**: 생성된 시나리오 코드(`code`)와 한글 설명(`translated_sentence`)이 포함된 객체. (없을 시 `null`)

## 🔄 3. 표준 대화 워크플로우

### **Step 1: 대화 시작 (기기 정보 전달)**
클라이언트는 첫 번째 메시지를 보낼 때 현재 연결된 기기 목록을 함께 보냅니다. 서버는 이 정보를 `session_id`에 귀속시켜 저장합니다.

```json
// Request
{
  "sentence": "거실 불 꺼줘",
  "session_id": "user_unique_abc",
  "connected_devices": { ... } // 현재 기기 목록 데이터
}
```

### **Step 2: 연속 대화 (메시지만 전달)**
두 번째 대화부터는 복잡한 기기 목록이나 이전 컨텍스트를 보낼 필요가 없습니다. 서버가 DB에서 `session_id`를 기준으로 모든 맥락을 복구합니다.

```json
// Request
{
  "sentence": "모든 불 꺼줘",
  "session_id": "user_unique_abc"
}
```

### **Step 3: 시나리오 최종 승인 및 등록**
사용자가 에이전트가 제안한 코드에 만족하여 "응" 또는 "yes"라고 답하면, 에이전트는 내부적으로 `add_scenario` 툴을 실행하여 최종 시나리오를 DB에 저장. (허브 컨트롤러 연결 필요)

---

## 💡
- **세션 관리**: 사용자가 로그아웃하거나 브라우저를 닫으면 `session_id`를 초기화해야 함.
- **UI 구현**: `last_result`가 존재할 때만 화면에 "코드 미리보기 공간"을 렌더링하고, 평소에는 일반 채팅창UI를 사용해야 함.
- **서버 영속성**: 대화 내역은 서버 DB(`data/joi.db`)에 보관되므로, 서버를 재시작해도 사용자 대화가 끊기지 않음.
