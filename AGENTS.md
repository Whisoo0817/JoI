# Joi Code: Specification and Generation API

This document defines the **Joi** automation language and the **JoI code-generation API** that turns a natural-language command into Joi code. Scope is limited to Joi itself and our generation pipeline/service вҖ” nothing else.

---

## 1. What is Joi Code?
Joi is a Domain-Specific Language (DSL) specifically designed for IoT automation. It is evaluated in a persistent, ticking environment on an IoT hub.

### вҡӣпёҸ 1.1 JSON Metadata (The Wrap)
A Joi automation is delivered as a JSON object containing metadata and the script itself.

```json
{
  "name": "morning_routine",
  "cron": "0 7 * * *",
  "period": 300000,
  "code": "if (any(#Light).Switch == true) { all(#Light).Off() }"
}
```
*   **`name`**: Scenario identifier. Auto-generated by the naming stage (re-translate вҶ’ short label; Korean input keeps a Korean label, spaces вҶ’ `_`).
*   **`cron`**: 5-field cron expression for scheduled triggers (e.g., `0 7 * * *` = 7 AM daily; `0 * * * *` = every hour on the hour). Empty string when the script is event-driven or manually triggered. Day-of-week is a digit `1=Mon вҖҰ 7=Sun`.
*   **`period` (milliseconds)**: Re-execution / polling interval.
    *   `0`: one-shot вҖ” the script runs once and ends (the common non-cyclic case).
    *   `N > 0`: the script re-executes every N ms to poll for state changes (e.g. `1000` = 1 Hz polling for sustained-condition and edge-trigger idioms).
*   **`code`**: The actual Joi logic string.

### вҡӣпёҸ 1.2 State Management: `:=` vs `=`
Joi scripts are executed repeatedly when `period > 0`.

*   **Initialize-once (`:=`)**: The right-hand side is evaluated **ONLY ONCE** at the first tick; the slot then persists across every tick. Use for state flags/counters: `triggered := false`, `hold_ticks := 0`.
*   **Per-tick assignment (`=`)**: Re-evaluated **EVERY TICK**. Use for fresh sensor reads and updates: `current = (#Light).Brightness`, `hold_ticks = hold_ticks + 1`.

### вҡӣпёҸ 1.3 Device Mapping: The Power of Tags
Devices are not targeted by hardcoded IDs in Joi. Instead, they are selected via **Tags** (e.g., `#Office`, `#Entrance`, `#Light`), resolved at runtime by the IoT hub.

*   **Single Selector `(#Tag)`**: A specific device instance matching the tag вҖ” `(#DeskLamp).On()`.
*   **Multi-tag `(#A #B)`**: **Intersection** вҖ” one device carrying BOTH tags. NOT a fan-out.
*   **Group Action `all(#Tag)`**: Executes on every device with the tag вҖ” `all(#Light).Off()`.
*   **Existence `any(#Tag)`**: True if *at least one* device satisfies the condition вҖ” `if (any(#Door).Contact == true)`.

### вҡӣпёҸ 1.4 Temporal and Logic Constructs
*   **`delay(N UNIT)`**: Non-blocking pause. Units: `MSEC`, `SEC`, `MIN`, `HOUR`. Example: `delay(5 MIN)`.
*   **`wait until (Condition)`**: Pause until a state/event holds вҖ” `wait until (any(#Motion).Motion == true)`.
*   **Logic / comparison**: `and` `or` `not` (never `&&` `||` `!`); `==` `!=` `>` `<` `>=` `<=`.
*   **Quantifier comparison (`==|`, `>=|`, вҖҰ)**: Collection operators paired with `all(...)` вҖ” `all(#Light).Brightness >=| 80` means "is *any* light вүҘ 80%". (`any(#X).Attr == v` is canonicalized to `all(#X).Attr ==| v`.)
*   **Time conditions read the `Clock` service**, never a `clock.time` string: `(#Clock).Hour >= 21`, `(#Clock).Weekday == "saturday"`.

## 2. Input Data: Connected Devices Format
Generation reads `connected_devices` to resolve phrases вҶ’ tags/categories.

```json
{
  "device_id_001": {
    "category": ["ContactSensor"],
    "tags": ["Entrance", "Door", "ContactSensor"]
  },
  "device_id_002": {
    "category": ["Switch", "Light"],
    "tags": ["Office", "MainLight", "Light", "Switch"]
  }
}
```
`nickname` may also be present (shown on the app card) and is used for name-based grounding ("мӮјм„ұ кіөкё°мІӯм •кё° нҒ°кұ°").

## 3. Joi Code Generation Pipeline (device-first, IR-mediated)
The active pipeline lives in `paper/run_local_ir.py` (`generate_joi_code`). It compiles a Korean/English command into a Joi block through a **Timeline IR**. LLM stages are interleaved with **deterministic Python** steps (grounding resolution, selector-tag minimization, quantifier choice) so small local models never decide what code can compute exactly.

### Stage 1 вҖ” Targeting
1.  **`device_retrieve`** (LLM, command only вҖ” does NOT see devices): parses the command into target groups, one per line: `role=<condition|read|action|notify> | by=label:<verbatim phrase>|channel:<speaker,toast> | scope=<all|any|one|auto>`.
2.  **`ground_targets`** (LLM, sees devices): maps each label phrase to a **criterion** (tag/category/nickname tokens; `+`=AND, `;`=OR-cluster) вҖ” not a raw device list.
3.  **`resolve_criterion` + `minimal_tags_for`** (Python): resolve the criterion to exact device sets and derive the tightest selector tag(s).
4.  **`device_resolve`** (LLM): pick the service per group and echo the given tags вҶ’ `(#Tag).Category.Method`. It does NOT choose the quantifier.
5.  **`quantifier_for`** (Python): add `all`/`any`/`one` deterministically from scope + role + match count.

### Stage 2 вҖ” Translation
The command is translated to English for the IR/lowering prompts; the original Korean is retained for human-facing arg text (Speaker/Toast/Email).

### Stage 3 вҖ” Resolve + IR extraction (parallel branches)
*   **Branch A**: `enum_cond_check` вҶ’ `enum_resolve` вҶ’ `arg_resolve` (fill argument values) вҶ’ `timeline_ir_extract` (the LLM emits the selector-free Timeline IR; validated against devices + catalog with a bounded retry).
*   **Branch B**: the precision selectors produced in Stage 1.
The IR is selector-free, so the two branches are independent.

### Stage 4 вҖ” Lowering (`joi_from_ir`)
The Timeline IR + precision selectors are mechanically lowered to a Joi block `{cron, period, script}`, routed by structural bucket (`noncycle` / `cycle`). Lowering is a lossless 1:1 translation вҖ” it adds no control flow not present in the IR.

### Stage 5 вҖ” Naming
`re_translate` (code вҶ’ English NL) вҶ’ `re_translate_kor` (вҶ’ Korean NL for Korean input) вҶ’ `scenario_name` (short label used as `name`).

**Timeline IR** is the pipeline's pivot: temporal/trigger logic is resolved into IR ops (`start_at`, `wait`, `cycle`, `if`, `call`, `read`, `delay`, `break`), then lowering picks the Joi idiom. See `paper/timeline_ir_extractor.md`, `files/joi_common.md`, `files/joi_noncycle.md`, `files/joi_cycle.md`.

## 4. Generation API (`app.py`, port 49999)
FastAPI service wrapping `generate_joi_code`. Prompts and the service catalog load once at import.

**`POST /generate_joi_code`** вҖ” request body:
```json
{
  "sentence": "л¶Ҳмқҙ мјңм ёмһҲмңјл©ҙ лӘЁл‘җ кәјмӨҳ",
  "model": "cyankiwi/Ornith-1.0-9B-AWQ-FP8",
  "connected_devices": { "...": { "category": [], "tags": [] } },
  "current_time": "2026-06-23T02:13:31",
  "other_params": null
}
```
Response is a `JoiLLMResponse` (`schemas.py`): `outcome`, `error_code`/`error_message`, `code` (list of `{name, cron, period, code}`), and `log` (`response_time`, `translated_sentence`, `logs` вҖ” the per-stage reasoning trace). Errors are typed (e.g. `device_not_connected`, `reasoning_failed`, `ir_catalog_*`).

Other endpoints: **`GET /health`**, **`POST /warmup`** (vLLM prefix-cache warmup).

**Backend LLM**: an OpenAI-compatible endpoint set by `LLM_BASE_URL` (default `http://localhost:8002/v1`); the model id is auto-discovered. Tuned for local 8вҖ“9B models (currently Ornith-1.0-9B / Qwen-9B class).

## 5. Design Notes
*   **Deterministic where it counts**: grounding resolution, selector-tag minimization, and quantifier choice are pure Python вҖ” the LLM never re-decides them, keeping output stable across runs.
*   **Context filtering (lost-in-the-middle)**: each stage sees only the device schemas/rules relevant to its targets (`files/devices/device_rules_<cat>.md`, section-scoped).
*   **Validation + bounded retry**: the IR is checked against connected devices and the service catalog; on violation the extract stage retries with a typed hint.
